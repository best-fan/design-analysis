# Git Commit Skill

生成符合项目规范的 git commit，确保提交前与远程仓库同步。

## 执行步骤

### 第一阶段：确认修改目录（必须先执行）

1. **检查仓库结构**
   - 运行 `git rev-parse --show-toplevel` 获取当前仓库根目录
   - 检查是否存在子模块：`git submodule status`
   - 检查是否存在子仓库（独立 .git 目录）：搜索子目录中的 `.git` 文件夹

2. **使用 AskUserQuestion 选择目标仓库**
   - 如果存在子仓库/子模块：列出所有仓库供用户选择
   - 选项示例：
     - "主仓库" - 在根目录操作
     - "子仓库A" - 进入指定子仓库操作
     - "所有仓库" - 依次处理所有仓库
   - 如果只有一个仓库：直接进入下一步

3. **切换到目标仓库目录**
   - 根据用户选择切换到对应目录
   - 使用 `cd <target-dir>` 进入子仓库
   - 确认当前工作目录为目标仓库

4. **运行 `git status` 查看未提交的文件**
   - 重点关注文件所在的目录结构
   - 分类整理：新增文件、修改文件、删除文件

5. **分析变更目录结构**
   - 按目录分组展示变更文件
   - 标识每个目录的变更类型（新增/修改/删除）
   - 估算每个目录的变更量

6. **使用 AskUserQuestion 展示目录变更概览**
   - 列出所有变更目录
   - 让用户选择要提交的目录范围
   - 选项示例：
     - "全部提交" - 提交所有变更
     - "按目录选择" - 选择特定目录
     - "手动选择文件" - 精确选择文件

7. **确认提交范围**
   - 根据用户选择确定最终要提交的文件列表
   - 展示确认后的文件清单
   - 用户确认后进入下一阶段

### 第二阶段：拉取远程代码（提交前必须执行）

8. **检查远程仓库状态**
   - 运行 `git remote -v` 确认远程仓库配置
   - 运行 `git fetch` 获取远程最新状态（不合并）

9. **检查本地与远程的差异**
   - 运行 `git status` 检查本地分支状态
   - 检查是否有远程新提交需要拉取

10. **使用 AskUserQuestion 询问是否拉取**
   - 如果远程有新提交：询问用户是否拉取远程代码
   - 如果远程无新提交：直接跳过拉取步骤

11. **执行拉取（用户确认后）**
   - 运行 `git pull` 拉取远程代码
   - **处理拉取结果**：
     - 成功无冲突：继续下一阶段
     - 有冲突：提示用户解决冲突后再继续

12. **冲突处理流程（如有冲突）**
   - 运行 `git status` 查看冲突文件列表
   - 展示冲突文件清单，按目录分组
   - 提示用户："检测到以下文件有冲突，请先解决冲突后再继续提交"
   - 使用 AskUserQuestion 让用户选择：
     - "查看冲突详情" - 展示冲突内容
     - "我已解决冲突" - 用户确认后继续
     - "取消提交" - 结束本次提交流程

### 第三阶段：生成 Commit 消息

13. 运行 `git diff --stat` 查看修改统计（仅针对确认的文件）
14. 运行 `git log --oneline -5` 查看最近的 commit 消息风格
15. 分析修改内容，生成 **多个候选 commit 消息**供用户选择
16. **使用 AskUserQuestion 展示候选消息，让用户选择确认**

### 第四阶段：执行 Git 操作

17. 用户确认后，使用 `git add` 添加确认的文件（仅添加用户选择的文件）
18. 使用 `git commit` 创建提交
19. **使用 AskUserQuestion 让用户确认是否推送远程**
20. 用户确认推送后，执行 `git push`

21. **多仓库处理（如选择了"所有仓库"）**
   - 如果用户选择了多个仓库，依次返回步骤1处理下一个仓库
   - 完成所有仓库提交后，展示提交汇总报告

## Commit 消息格式

```
<type>(<scope>): <subject>

<body>

Authored-By: Claude Code AI
```

## Commit 类型

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复问题 |
| docs | 文档变更 |
| style | 代码格式调整（不影响功能） |
| refactor | 重构代码 |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具相关 |

## Scope 命名规范

Scope 应对应变更的主要目录或模块：

| 目录 | Scope 示例 |
|------|------------|
| `src/components/` | `ui`, `components` |
| `src/routes/<page>/` | `<page-name>` |
| `src/stores/` | `store`, `state` |
| `src/hooks/` | `hooks` |
| `src/http/` | `api`, `http` |
| `.claude/skills/` | `skill`, `<skill-name>` |

## 重要约束

### 必须先确认目标仓库

- **禁止跳过仓库选择步骤**
- 必须先检查是否存在子仓库/子模块
- 如果存在多个仓库，必须让用户选择目标仓库
- 确认目标仓库后才能进入下一阶段

### 必须先确认目录

- **禁止跳过目录确认步骤**
- 必须先展示变更目录概览
- 必须让用户明确选择提交范围
- 确认后才能进入下一阶段

### 提交前必须拉取（如有远程仓库）

- **禁止在有远程新提交时跳过拉取步骤**
- 必须先运行 `git fetch` 检查远程状态
- 如果远程有新提交，必须询问用户是否拉取
- 拉取成功后才能继续执行 `git add`
- 有冲突时必须让用户解决后再继续
- 无远程仓库或无新提交时可跳过拉取步骤

### Git 操作规范

- 使用 HEREDOC 格式传递 commit 消息，避免换行问题
- commit 消息末尾添加 `Authored-By` 标识
- 只 commit 用户明确确认的文件
- 不要使用 `--no-verify` 跳过 hooks
- 不要使用 `-A` 或 `.` 添加所有文件，要明确指定文件
- 不要使用 `--force` 强制推送，除非用户明确要求并理解风险

### 文件过滤

以下文件类型默认不提交（需用户明确指定才能提交）：

- `.env`, `.env.local`, `.env.*.local` - 环境配置
- `*.log` - 日志文件
- `.DS_Store` - macOS 系统文件
- `node_modules/` - 依赖目录
- `dist/`, `build/` - 构建产物
- `*.zip`, `*.tar.gz` - 压缩包
