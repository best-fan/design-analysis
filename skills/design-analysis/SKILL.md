# 设计稿分析

## 使用时机

在以下情况使用此技能：

- 分析设计稿（`.pen`、Figma 链接、MasterGo 链接、其它设计图或标注），梳理界面结构、样式、元素
- 分析本地图片（设计稿截图、UI 图片），通过视觉识别提取信息
- 产出 UI 分析清单文档，供开发实现或验收对照

不限定阶段：可在写提案前、提案中、或单独进行「只分析不写提案」。

---

## 核心原则

**按「从上到下、从左到右、从外到里」顺序分析，逐条准确记录文字、图片、布局、层级四类重中之重。**

**输出后必须检查：区域顺序与设计稿一致、布局数据精准无误。若存在问题，必须补充和修正。**

详见：
- `references/analysis-order.md` - 分析顺序（必守）
- `references/analysis-priorities.md` - 四类重中之重

## ⚠️ 强制执行机制

> **规范写得再清楚，如果执行者没有建立强制执行机制，就会被绕过。**

**必须遵守 `references/enforcement-mechanism.md` 中的强制执行机制：**

### 禁止行为

| 禁止行为 | 后果 | 正确做法 |
|----------|------|----------|
| ❌ 凭经验推断图表类型 | 图表类型错误 | ✅ 读取节点 name 字段确认实际类型 |
| ❌ 凭经验推断子元素归属 | 子元素归属错误 | ✅ 用 MCP 工具读取实际 children 数据 |
| ❌ 跳过「用 MCP 读取子节点」步骤 | 遗漏或虚构子元素 | ✅ 每个区域都必须读取 |
| ❌ 跳过校验一到校验十 | 错误未被发现 | ✅ 逐项执行校验 |

### 强制检查清单

**在分析过程中，必须逐项完成以下检查清单并打勾：**

```markdown
## 分析前强制检查
- [ ] 已获取设计稿整体尺寸
- [ ] 已统计顶层区域数量
- [ ] **我承诺：不凭经验推断图表类型**
- [ ] **我承诺：每个区域都用 MCP 工具读取实际子节点**

## 每个区域分析时强制检查
- [ ] **已使用 MCP 工具读取该区域的实际 children 数据**
- [ ] **已读取图表节点的 name 字段确认图表类型**（禁止推断）
- [ ] **已按实际节点树记录层级结构**（禁止凭视觉推断）

## 输出前强制检查
- [ ] **我确认：没有凭经验推断任何图表类型**
- [ ] **我确认：每个区域都用 MCP 工具读取了实际子节点**
- [ ] **我确认：没有虚构任何子元素或模块**
```

详见 `references/enforcement-mechanism.md`。

## 目标产出

| 产出物 | 路径 | 用途 |
|--------|------|------|
| UI 分析清单 | `openspec/ui-checklist/{序号}-{模块名}.md` | 开发时精确还原；验收时对照基准 |

### 示例文件

参考示例：`assets/example-ui-checklist.md`

该示例展示了一个完整的「项目结算看板」UI 分析清单，包含所有九个章节的标准格式和示例数据，可作为创建新清单时的参考模板。

### 命名约束（必须遵守）

| 约束项 | 规范 | 示例 |
|--------|------|------|
| 序号 | **三位数字递增，不可重复** | `001`、`002`、`003`...`010` |
| 模块名 | 使用英文，kebab-case | `login-page`、`dashboard-header` |
| 文件名 | **`{序号}-{模块名}.md`** | 序号与模块名组合 |

#### 文件创建规则

**步骤 1：获取现有文件命名**
- 查看 `openspec/ui-checklist/` 目录下已有的文件
- 提取所有文件名中的序号部分（前三位数字）

**步骤 2：确定新序号**
- 若目录为空（首次分析）：新序号为 `001`
- 若目录非空：取现有最大序号 + 1
  - 示例：现有 `001-login-page.md`、`002-dashboard-header.md`，则新序号为 `003`

**步骤 3：创建文件**
- 文件命名格式：`{序号}-{模块名}.md`，例如 `003-user-profile.md`
- 文件路径：`openspec/ui-checklist/{序号}-{模块名}.md`

**序号复用规则：**
- 仅当**最后一个序号**被删除时，该序号可复用
- 中间序号被删除后**永久空置**，不可复用

**示例场景：**
| 操作 | 现有文件 | 新文件序号 |
|------|---------|-----------|
| 首次创建 | （空） | `001` |
| 第二次创建 | `001-login-page.md` | `002` |
| 第三次创建 | `001-login-page.md`、`002-dashboard-header.md` | `003` |
| 删除 `002-dashboard-header.md`（中间） | `001-login-page.md`、`003-user-profile.md` | `004`（跳过 002） |
| 删除 `003-user-profile.md`（最后） | `001-login-page.md`、`002-dashboard-header.md` | `003`（复用） |


**正确示例：**
- `openspec/ui-checklist/001-login-page.md`
- `openspec/ui-checklist/002-dashboard-header.md`
- `openspec/ui-checklist/003-user-profile.md`
- `openspec/ui-checklist/010-settings.md`

**错误示例：**
- `openspec/ui-checklist/1-登录页.md`（序号非三位、模块名为中文）
- `openspec/ui-checklist/001-登录.md`（模块名为中文）
- `openspec/ui-checklist/001-login/`（路径错误，应为文件而非文件夹）
- `openspec/ui-checklist/001-new-module.md`（序号已被占用）

## 必须遵守
- 不允许修改已存在的文件
- 必须创建新的文件
- **必须严格按照 `references/output-analysis-checklist.md` 中的 UI 分析清单文档模板生成文档**，包含所有章节：
  1. 文档说明（设计稿路径/链接、分析日期）
  2. 一、设计稿整体分析（页面状态、整体尺寸、区域列表表格、区域间间距）
  3. 二、区域详细分析（按文字/图片/布局/层级格式，每个区域独立章节）
  4. 三、样式规范汇总（颜色、字体、圆角、间距、阴影）
  5. 四、交互状态分析（默认/hover/active/禁用等状态）
  6. 五、占位元素说明（图标占位尺寸与位置、图片占位尺寸与比例）
  7. 六、常见遗漏检查点（checklist 格式）
  8. 七、实现建议（组件划分、样式文件、开发顺序、常见错误）
  9. 八、验证检查清单（含校验表格和最终清单）
  10. 九、备注（坐标系统、响应式、占位策略等）
- **不得遗漏任何模板章节，不得使用自创格式**

## 工作流程

根据设计稿类型选择分析路径：

### 路径 A：结构化设计稿（.pen / Figma / MasterGo）

1. **建立布局 Map** - 获取设计稿结构、页面状态、整体尺寸、区域划分
   - 详见 `references/workflow-layout-map.md`
   - 工具使用详见 `references/tools-design-guidelines.md`

2. **区域与元素提取** - 对每个区域按「从外到里」逐项提取
   - 详见 `references/workflow-element-extraction.md`

3. **样式规范汇总** - 汇总颜色、字体、圆角、间距、阴影等
   - 详见 `references/workflow-style-summary.md`

4. **输出 UI 分析清单** - **必须严格按照模板输出文档**
   - **模板路径**：`references/output-analysis-checklist.md`
   - **禁止遗漏章节**：必须包含模板中的所有章节（一~九），不得使用自创格式
   - **禁止修改格式**：章节标题、内容结构必须与模板完全一致
   - 质量检查详见 `references/workflow-output-checklist.md`
   - **生成后必须执行校验**：详见 `references/output-analysis-checklist.md`「生成后校验」
     - **校验前必须重新获取设计稿数据**：使用对应 MCP 工具重新获取设计稿结构和节点信息，确保校验基于最新数据
     - **⚠️ 校验零：区域数量校验（必须首先执行）**
       - **统计设计稿顶层节点数量**：使用设计稿工具遍历所有直接子节点
       - **统计文档区域列表行数**
       - **数量对比**：确保设计稿区域数 = 文档记录数
       - 若不一致，**必须立即查找并补充遗漏区域**
     - 校验一：区域顺序校验（从上到下、从左到右）
     - 校验二：区域详细分析顺序校验
     - **校验三：标题文字一致性校验（必须严格执行）**
       - 将清单中记录的标题文字与设计稿实际内容逐条对比
       - 若发现不一致（包括多字、少字、错别字、用词差异），**必须立即修正**
       - 修正完成后再次校验，直至完全一致
     - **校验四：模块存在性验证（必须严格执行）**
       - 将清单中记录的所有模块在设计稿中逐一验证是否存在
       - 若发现虚构模块（文档有但设计稿无），**必须立即删除**
       - 修正完成后再次校验，确保所有模块均真实存在
     - **校验五：区域列表一致性校验（必须严格执行）**
       - 逐区域核对设计稿实际内容
       - 区域标题与设计稿实际标题完全一致
       - 修正完成后再次校验
     - **校验六：表格列标题专项校验（必须严格执行）**
       - **逐列读取**设计稿表格的实际列标题文字
       - **禁止根据经验推断**列名（如假设有"项目编号、项目名称"等）
       - 若发现不一致，**必须立即修正**
       - 修正完成后再次校验
     - **校验七：区域列表包含子元素一致性校验（必须严格执行）**
       - **逐区域提取**设计稿实际子元素（文字、图标、按钮、输入框等）
       - **识别复合组件**（如统计卡片组、筛选条件组）
       - 与「区域列表」表格中的「包含子元素」列对比
       - 若发现遗漏或描述不准确，**必须立即修正**
       - 修正完成后再次校验
     - **校验八：子元素排序校验（必须严格执行）**
       - 识别区域内所有横向排列的子元素组
       - 提取每个子元素的 x 坐标
       - 按 x 坐标排序确认视觉顺序
       - 文档记录顺序与设计稿实际顺序对比
       - 若发现不一致，**必须立即修正**
       - 修正完成后再次校验
     - **⚠️ 校验九：子元素归属校验（必须严格执行）**
       - **使用 MCP 工具读取每个区域的实际子元素数据**
       - **计算每个区域的边界范围**（x, y, width, height）
       - **验证每个子元素坐标是否在父区域范围内**
       - 若发现归属错误（子元素坐标超出父区域边界），**必须立即修正或删除**
       - 若发现虚构子元素（文档有但设计稿无），**必须立即删除**
       - 修正完成后再次校验
    - **⚠️ 校验十：层级结构校验（必须严格执行）**
       - **使用 MCP 工具逐区域提取实际节点树**
       - **将文档记录的层级结构与设计稿实际节点树逐层对比**
       - **核对项目**：
         - 子节点数量是否一致
         - 子节点名称/类型是否一致
         - 子节点坐标是否准确
         - 嵌套关系是否正确
       - **禁止凭视觉推断层级**（如"图例应该在标题下面"）
       - 若发现层级不一致，**必须立即修正**
       - 修正完成后再次校验
     - **若校验不通过，必须修正后再进入下一步**

5. **区域检查与对比完善** - 按模块区域逐一检查对比，完善遗漏和不准确之处
   - 详见 `references/workflow-module-verification.md`
   - **检查前必须重新获取设计稿数据**：使用对应 MCP 工具重新获取设计稿结构，基于最新数据进行对比检查
   - **核心检查项（必须执行）**：
     - **模块存在性验证**：文档中每个模块必须能在设计稿中找到对应区域，**不得虚构不存在的模块**
     - **区域顺序一致性**：文档中区域排列顺序必须与设计稿从上到下、从左到右顺序一致
     - **布局数据精准性**：位置坐标 (x, y)、尺寸 (宽, 高)、间距 (padding, gap, margin) 必须与设计稿精确匹配
     - **标题文字一致性**：标题文字必须与设计稿完全一致
   - **若存在问题，必须补充和修正**

### 路径 B：本地图片（设计稿截图 / UI 图片）

1. **图像识别** - 使用视觉能力读取图片，提取页面结构、文字、布局、样式
   - 分析顺序遵守 `references/analysis-order.md`
   - 详见 `references/tools-design-guidelines.md`「本地图片处理」

2. **结构化整理** - 将识别信息按区域整理，建立布局 Map
   - 参考 `references/workflow-layout-map.md`

3. **样式规范汇总** - 提取颜色、字体、圆角、间距等
   - 详见 `references/workflow-style-summary.md`

4. **输出 UI 分析清单** - **必须严格按照模板输出文档**
   - **模板路径**：`references/output-analysis-checklist.md`
   - **禁止遗漏章节**：必须包含模板中的所有章节（一~九），不得使用自创格式
   - **禁止修改格式**：章节标题、内容结构必须与模板完全一致

5. **区域检查与对比完善** - 按模块区域逐一检查对比，完善遗漏和不准确之处
   - 详见 `references/workflow-module-verification.md`
   - **检查前必须重新获取设计稿数据**：使用视觉识别能力重新读取图片，基于最新识别结果进行对比检查
   - **核心检查项（必须执行）**：
     - **模块存在性验证**：文档中每个模块必须能在设计稿中找到对应区域，**不得虚构不存在的模块**
     - **区域顺序一致性**：文档中区域排列顺序必须与设计稿从上到下、从左到右顺序一致
     - **布局数据精准性**：位置坐标 (x, y)、尺寸 (宽, 高)、间距 (padding, gap, margin) 必须与设计稿精确匹配
     - **标题文字一致性**：标题文字必须与设计稿完全一致
   - **若存在问题，必须补充和修正**
