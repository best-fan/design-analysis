## 第一步：建立布局 Map

### 目标

获取设计稿的整体结构，确定页面状态、整体尺寸、区域划分和区域间关系。

### 🔴🔴🔴 强制步骤：记录区域边界值（必须首先执行）

**在建立布局 Map 时，必须立即记录每个区域的边界值！**

**此步骤不可跳过！后续所有元素归属判断都依赖此步骤！**

```markdown
## 区域边界验证表（分析阶段必填，写入 memory）

| 区域名称 | x | width | 左边界 | 右边界 |
|----------|---|-------|--------|--------|
| 区域1 | ___ | ___ | ___ | ___ |
| 区域2 | ___ | ___ | ___ | ___ |
| 区域3 | ___ | ___ | ___ | ___ |
```

**为什么必须执行此步骤？**

| 不执行此步骤的后果 | 真实案例 |
|------------------|----------|
| 元素归属错误 | 年份选择器 x=546 被错误归类到右侧区域 |
| 文档层级错误 | 层级结构与设计稿实际不一致 |
| 前端开发错误 | 组件位置实现错误 |

### 强制步骤：统计顶层节点数量

**在建立布局 Map 之前，必须执行以下步骤**：

1. 获取设计稿根节点
2. 遍历根节点的所有直接子节点
3. 过滤出具有实际尺寸（w > 100, h > 50）的 GROUP/FRAME/INSTANCE 节点
4. 统计总数并记录
5. 按 y 坐标排序后逐一命名

### 必须记录的内容

| 项目 | 说明 |
|------|------|
| 页面状态 | 若有多个 frame，分别标注 id 与用途 |
| 整体尺寸 | 宽度、高度（如 1440×1150） |
| 区域列表 | 按从上到下（y）排序，记录名称、x/y/w/h、包含的子元素 |
| 区域间间距 | 上一区域底部到下一区域顶部的垂直间距 |

### ⚠️ 区域命名规则（必须严格执行）

**区域名称必须与区域内文字标题一致！**

| 场景 | 正确做法 | 错误做法 |
|------|----------|----------|
| 区域内有标题文字 | 区域名称 = 标题文字内容 | 区域名称 = 「XX卡片」「XX区域」 |
| 区域内无标题文字 | 根据功能命名，如「顶部导航栏」 | 使用模糊名称 |

**示例**：
- ✅ 正确：区域名称「项目结算总额」，标题文字「项目结算总额」
- ❌ 错误：区域名称「项目结算总额卡片」，标题文字「项目结算总额」

### ⚠️ 区域划分规则（必须严格执行）

**独立卡片应作为独立区域，而不是"左右布局"的子区域！**

| 场景 | 正确做法 | 错误做法 |
|------|----------|----------|
| 左右两个独立卡片 | 分别作为区域2、区域3 | 合并为一个"左右布局"区域 |
| 左侧独立卡片 + 右侧包含多元素 | 左侧为独立区域，右侧区域包含其子元素 | 强行合并为"左右布局" |

**判断标准**：
1. **视觉独立性**：卡片是否有独立的背景、边框、圆角？
2. **功能独立性**：卡片是否表达独立的数据/功能？
3. **标题独立性**：卡片是否有独立的标题？

**示例**：
```
✅ 正确划分：
- 区域2：项目结算总额（独立卡片）
- 区域3：项目营收平衡情况（包含标题、Tab、数值卡片、图表）

❌ 错误划分：
- 区域2：数据概览区域（左右布局）
  - 左侧：项目结算总额卡片
  - 右侧：项目营收平衡情况
``` |

## 第二步：区域与元素提取

### 强制前提：必须使用工具读取实际子元素

**禁止凭视觉猜测或凭经验推断子元素归属！**

### ⚠️ 强制要求：读取深度设置

**必须使用足够的读取深度，否则会遗漏嵌套内容！**

```javascript
// ✅ 正确：读到 depth 4，展开实例
batch_get({
  nodeIds: ["target-node"],
  readDepth: 4,
  resolveInstances: true,
  resolveVariables: true
})

// ❌ 错误：只读到 depth 2，可能遗漏第二层嵌套
batch_get({ nodeIds: ["target-node"], readDepth: 2 })
```

| 设计复杂度 | 推荐 readDepth | 说明 |
|------------|----------------|------|
| 简单页面 | 2 | 单层结构 |
| 中等复杂 | 3 | 有嵌套容器 |
| 复杂页面 | 4+ | 多层嵌套、实例组件 |

### ⚠️ 强制要求：组件状态字段检查

**分析阶段必须检查组件状态字段（enabled/visible/checked），否则会误判组件可见性！**

| 字段 | 值 | 状态说明 | 对设计稿的影响 |
|------|-----|----------|----------------|
| `enabled` | `false` | 禁用状态 | **组件不显示或不可交互** |
| `enabled` | `true` | 启用状态 | 正常显示 |
| `visible` | `false` | 隐藏状态 | **组件不可见** |
| `checked` | `true/false` | 选中状态 | checkbox 的选中状态 |

### ⚠️ 真实遗漏案例（2026-04-07）

**案例：表格禁用列被误认为可见**

| 问题 | 分析阶段处理 | 设计稿实际 |
|------|--------------|------------|
| Checkbox列 | 未检查 enabled 字段 | `enabled: false`，不可见 |
| 操作列 | 未检查 enabled 字段 | `enabled: false`，不可见 |

**后果**：文档记录了不可见的列，导致前端开发实现不需要的功能

### 状态字段检查范围

| 组件类型 | 必须检查的字段 | 分析阶段处理 |
|----------|----------------|--------------|
| 表格列 | `enabled` | enabled: false → 不在文档中记录 |
| 按钮 | `enabled` | enabled: false → 标注「禁用状态」 |
| Checkbox | `enabled`, `checked` | 记录禁用和选中状态 |
| Tab 项 | `enabled` | enabled: false → 不在文档中记录 |
| 输入框 | `enabled`, `visible` | 记录禁用和可见状态 |

### 强制规则（分析阶段必须执行）

- [ ] **读取表格列时，必须检查每列的 enabled 字段**
- [ ] **enabled: false 的列，不在文档中记录**
- [ ] **读取按钮时，必须检查 enabled 字段**
- [ ] **读取 Checkbox 时，必须检查 enabled 和 checked 字段**
- [ ] **不虚构不存在的元素**（如「类型」列、「进度」列）

### ⚠️ 警惕：嵌套容器同名问题

**父子节点或兄弟节点可能同名，必须用坐标区分！**

```
// 危险示例：Group 220 包含另一个 Group 220
Group 220（基础信息区）
├── Group 219（第一行，y=20）
└── Group 220（第二行，y=94）← 名字相同，容易被忽略！
```

**解决方法**：
1. 检查所有子节点的 y 坐标
2. 如果 y 坐标跳跃 > 20px，检查是否有多行内容
3. 统计子节点数量，与文档对比

### ⚠️ 强制步骤：子元素坐标归属验证（分析阶段必执行）

**规范写得再清楚，如果没有强制执行机制，就会被绕过！**

**🔴🔴🔴 此步骤依赖第一步的「区域边界验证表」，必须先完成第一步！**

#### 真实错误案例（2026-04-08）

**案例：年份选择器归属错误**

| 问题 | 分析时操作 | 设计稿实际 |
|------|------------|------------|
| 年份选择器 | 凭视觉直觉判断属于区域3 | x=546，通过坐标验证属于区域2 |

**错误过程**：
1. 读取区域结构 → 看到 y2Xfx 节点
2. 视觉直觉判断 → "年份选择器在右上角附近，应该控制右侧图表"
3. 直接写入文档 → 没有验证 x=546 是否在区域内

**正确过程**：
1. **先完成第一步** → 记录区域边界：区域2右边界=682，区域3左边界=698
2. 读取元素坐标 → y2Xfx: x=546
3. **验证归属** → 160 ≤ 546 < 682 ✅ 属于区域2
4. 写入文档 → 年份选择器归入区域2

#### 🔴 强制检查清单（每个元素必须验证）

**在将任何元素写入文档之前，必须验证：**

- [ ] **已从第一步获取区域边界值**
- [ ] **已读取元素的 x 坐标**
- [ ] **已验证 x 坐标在哪个区域边界内**
- [ ] **边界附近的元素（x 在边界±50px）已重点验证**

#### 验证公式

```
元素归属正确的条件：
  区域.x ≤ 元素.x < 区域.x + 区域.width

或简化判断：
  元素.x >= 区域左边界 AND 元素.x < 区域右边界
```

#### ⚠️ 真实遗漏案例（2026-04-08）

**案例：年份选择器归属错误**

| 元素 | 实际坐标 | 错误归属 | 正确归属 |
|------|----------|----------|----------|
| 年份选择器 | x=546 | 右侧图表区 | 左侧卡片区 |

**验证**：
- 左侧卡片区：x=160, 右边界=160+522=682 → 160 ≤ 546 < 682 ✅
- 右侧图表区：x=698 → 546 < 698 ❌

**原因**：凭视觉直觉认为年份选择器控制右侧图表，未验证坐标边界

**教训**：
1. **严禁凭视觉判断元素归属**
2. **每个元素必须验证 x 坐标是否在区域内**
3. **左右布局尤其要检查边界附近的元素**

#### 强制规则

- [ ] **记录区域时必须同时记录边界值（左边界、右边界）**
- [ ] **每个元素归属必须验证 x 坐标是否在区域边界内**
- [ ] **左右布局中，边界附近（x 在边界±50px 范围）的元素必须重点验证**
- [ ] **层级结构中，必须标注独立节点的坐标位置**
- [ ] **发现坐标归属问题后，必须立即修正文档中的层级结构，确保文档与设计稿一致**

#### ⚠️ 真实修正案例（2026-04-08）

**案例：发现坐标归属问题后修正层级结构**

| 阶段 | 操作 | 说明 |
|------|------|------|
| 初次分析 | 年份选择器写入右侧区域层级 | 凭视觉直觉判断 |
| 坐标验证 | x=546，验证 160≤546<682 | 发现属于左侧区域 |
| **立即修正** | 移除右侧层级，添加到左侧层级 | 修正文字、图片、层级三个部分 |

**修正内容**：
1. **文字部分**：将年份选择器文字从区域3移到区域2
2. **图片部分**：将年份选择器图标从区域3移到区域2
3. **层级结构**：从区域3层级中删除，添加到区域2层级中

**教训**：
- 发现问题后**立即修正**，不要等到校验阶段
- 修正时要**全面修正**：文字、图片、层级三个部分都要修改
- 在层级中**添加备注**：说明该元素的坐标归属依据

### 核心原则：按坐标排序

**DSL 树结构遍历顺序 ≠ 视觉布局顺序**

所有横向排列的子元素，必须按 x 坐标从小到大排序后记录。

### ⚠️ 左右布局专项分析（必须严格执行）

**左右分栏布局是高频场景，必须专项分析宽度比例和高度关系！**

常见左右布局场景：

| 场景 | 左侧区域 | 右侧区域 | 分析重点 |
|------|----------|----------|----------|
| 导航栏 + 内容区 | 固定宽度导航栏 | 自适应宽度内容区 | 左侧固定宽度、右侧撑满 |
| 筛选区 + 表格区 | 固定宽度筛选区 | 自适应宽度表格区 | 左侧固定宽度、右侧撑满 |
| 列表 + 详情 | 固定宽度列表 | 自适应宽度详情 | 左侧固定宽度、右侧撑满 |
| 左右卡片布局 | 左卡片 | 右卡片 | 左右宽度比例、间距 |

### 左右布局必须记录的内容

```markdown
### 左右布局分析清单

| 项目 | 左侧区域 | 右侧区域 | 说明 |
|------|----------|----------|------|
| 区域名称 | ___ | ___ | 如：导航栏/内容区 |
| x 坐标 | ___ px | ___ px | 左侧通常从 0 或页面边距开始 |
| y 坐标 | ___ px | ___ px | 通常相同（同一行） |
| 宽度 | ___ px | ___ px | 左侧固定？右侧自适应？ |
| 高度 | ___ px | ___ px | 是否撑满？是否相同？ |
| 宽度比例 | ___ : ___ | 如 240 : 1200（1:5） |
| 左右间距 | ___ px | 左右区域之间的间距 |
| 垂直对齐 | ___ | top / center / bottom / stretch |
```

### 左右布局强制规则

- [ ] **必须记录左右区域的宽度比例**（如 240:1200 = 1:5）
- [ ] **必须标注左侧宽度是否固定**（固定 px / 自适应 %）
- [ ] **必须标注右侧宽度是否自适应**（撑满剩余空间）
- [ ] **必须记录左右区域的垂直对齐方式**
- [ ] **必须记录左右区域之间的间距**

### ⚠️ 左右布局常见错误

| 错误类型 | 问题描述 | 正确做法 |
|----------|----------|----------|
| 未区分固定/自适应 | 左侧导航栏宽度记录为百分比 | 记录左侧固定宽度（如 240px） |
| 未记录宽度比例 | 只记录各自宽度，不计算比例 | 计算并记录比例（如 1:5） |
| 未记录垂直对齐 | 左右区域高度不同但未说明 | 记录对齐方式（stretch/center） |
| 未记录间距 | 左右区域紧邻但未记录间距 | 记录 gap 值（如 16px） |

### 四类必记内容

#### 层级（强制要求）

**每个区域必须包含 `#### 层级` 小节，记录节点树结构！**

**禁止凭视觉推断层级！必须使用工具读取实际节点树！**

记录格式：
```markdown
#### 层级

```
容器节点 (ID)
├── 子节点1 (ID) - 说明
├── 子节点2 (ID) - 说明
│   ├── 孙节点1 (ID) - 说明
│   └── 孙节点2 (ID) - 说明
└── 子节点3 (ID) - 说明
```
```

**层级记录必须包含**：
- 节点名称
- 节点 ID
- 节点类型（Group/Frame/Rectangle/Text 等）
- 简要说明

**⚠️ 层级记录排除规则（必须严格执行）**：

**`enabled: false` 或 `visible: false` 的元素不显示在层级结构中！**

| 元素状态 | 是否显示在层级结构 | 说明 |
|----------|-------------------|------|
| `enabled: false` | ❌ 不显示 | 组件禁用，设计稿中不可见 |
| `visible: false` | ❌ 不显示 | 组件隐藏，设计稿中不可见 |
| `enabled: true` 或无此字段 | ✅ 显示 | 正常显示 |

**示例**：
```
❌ 错误写法：
├── 表格
│   ├── Checkbox列 - enabled: false, 不显示
│   ├── 项目名称列
│   └── 操作列 - enabled: false, 不显示

✅ 正确写法：
├── 表格
│   └── 项目名称列
```

**原因**：
- 层级结构是开发实现的参考依据
- `enabled: false` 的元素不会被实现，不应出现在层级中
- 避免开发误解，实现不存在的功能

#### 文字

每处文字需记录：内容、字体、字号、字重、颜色、行高。

**易漏点**：欢迎语/大标题、副标题、按钮/链接文案、占位提示、Tab 文字。

#### 图片

每处图片需记录：尺寸、位置、比例、圆角、间距、多图拼接布局。

**易漏点**：小图标、占位图、多图拼接布局、圆角和比例。

#### 布局

记录项：位置（x, y）、尺寸（width, height）、间距（padding, gap, margin）、对齐方式、排列方向。

**易漏点**：间距方向性、对齐方式、排列方向。

#### 层级

**必须使用工具读取实际节点树，禁止凭视觉推断！**

记录项：嵌套关系、先后顺序、叠放关系。

### 表格区域专项

表格是**最高频错误区域**，必须严格按照以下步骤分析：

#### ⚠️⚠️⚠️ 真实遗漏案例（2026-04-08）

**案例：遗漏「项目名称」列**

| 问题 | 原因 | 后果 |
|------|------|------|
| 「项目名称」列未记录 | 只凭部分数据推断，未系统遍历所有列 | 文档列数错误，前端开发遗漏必要功能 |

**错误过程**：
1. 读取表格列节点 → 看到 10 个列节点
2. 部分表头内容是占位符「Title」→ 凭经验跳过
3. 未逐列读取每个表头的实际内容 → 遗漏「项目名称」列

**正确过程**：
1. 读取表格列节点 → 获取所有列的 children 列表
2. **逐列遍历** → 对每个列节点执行：
   - 检查列节点的 `enabled` 字段
   - 读取列节点的第一个子节点（表头单元格）
   - 读取表头单元格内的文字节点内容
3. **记录所有列** → 不管内容是占位符还是实际文字

#### 🔴🔴🔴 强制步骤：表格列系统性遍历

**必须按以下步骤逐一分析，禁止跳过任何列！**

```markdown
### 表格列分析清单（分析阶段必填）

步骤1：获取表格 columns 节点的所有子节点
- 列节点总数：___ 个

步骤2：逐列遍历（每个列节点必须执行）
| 列序号 | 列ID | enabled | 表头内容 | 列宽 | 是否记录 |
|--------|------|---------|----------|------|----------|
| 1 | ___ | true/false | ___ | ___ px | ✅/❌禁用 |
| 2 | ___ | true/false | ___ | ___ px | ✅/❌禁用 |
| ... | ... | ... | ... | ... | ... |

步骤3：验证列宽总和
- 列宽总和：___ px
- 表格宽度：___ px
- 差值：___ px（应接近0）
```

#### 强制规则（违反即错误）

- [ ] **必须获取表格 columns 节点的完整 children 列表**
- [ ] **必须对每个列节点逐一读取，禁止凭经验跳过**
- [ ] **必须读取每个列节点的表头单元格内文字，不能凭列名推断**
- [ ] **表头内容是占位符（如「Title」）时，仍需记录该列存在**
- [ ] **必须检查每个列节点的 `enabled` 字段，不是表头单元格的字段**
- [ ] **必须验证列宽总和 ≈ 表格宽度**
- [ ] **文档列数必须与 enabled: true 的列数完全一致**

#### 分析步骤

1. **获取列节点列表**：读取表格 columns 节点的所有 children
2. **统计列总数**：记录列节点总数
3. **逐列遍历**：对每个列节点执行：
   - 检查列节点的 `enabled` 字段
   - 读取列节点的 width 属性
   - 读取第一个子节点（表头单元格）
   - 读取表头单元格内的文字节点
4. **验证数量**：列宽总和应接近表格宽度
5. **筛选可见列**：最终文档只记录 `enabled: true` 或无 enabled 字段的列

#### 常见错误模式

| 错误模式 | 表现 | 正确做法 |
|----------|------|----------|
| 凭经验跳过 | 看到占位符「Title」就跳过 | 必须记录该列存在 |
| 部分读取 | 只读取了部分列的表头 | 必须遍历所有列节点 |
| 字段误判 | 检查表头单元格的 enabled 而非列节点 | 检查列节点的 enabled 字段 |
| 遗漏验证 | 不验证列宽总和 | 列宽总和应接近表格宽度 |

### ⚠️ 表格列状态字段检查（分析阶段必须执行）

**分析阶段**需要检查每列的 `enabled` 字段，但**最终文档只记录可见列**：

## 第四步：输出文档

### 文档路径

```
openspec/ui-checklist/{序号}-{模块名}.md
```

### 序号规则

- 三位数字递增（001, 002, 003...）
- 新序号 = 现有最大序号 + 1

### 文档结构（九大章节）

5. **占位元素说明**：图标、图片占位
6. **常见遗漏检查点**：已核对的检查项
7. **实现建议**：组件划分、样式文件、开发顺序
8. **验证检查清单**：后续验收用检查项

## 第五步：区域检查与对比完善

### 核心检查项（必须执行）

1. **区域顺序一致性**：文档顺序与设计稿从上到下、从左到右一致
2. **布局数据精准性**：位置、尺寸、间距精确匹配
3. **标题文字一致性**：标题文字完全一致
4. **模块存在性验证**：所有模块均存在于设计稿中

### 检查前必须重新获取设计稿数据

**禁止仅凭记忆或已提取的缓存数据进行检查！**

### 区域顺序检查

1. 获取设计稿区域坐标
2. 按 y 排序，y 相同时按 x 排序
3. 对比文档顺序

### 模块存在性验证

1. 提取文档中所有模块名称
2. 使用工具逐一验证每个模块是否真实存在
3. 删除虚构模块

### 布局数据检查

| 数据类型 | 检查项 | 精度要求 |
|----------|--------|----------|
| 位置坐标 | x, y | 精确到像素 |
| 尺寸 | 宽度、高度 | 精确到像素 |
| 内边距 | padding | 精确到像素 |
| 间距 | gap | 精确到像素 |

### 层级结构检查

**必须使用工具读取实际节点树**

- 子节点数量是否一致
- 子节点名称/类型是否一致
- 嵌套关系是否正确
