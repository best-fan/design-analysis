# 第五步：区域检查与对比完善

## 概述

第五步是在 UI 分析清单文档生成后，按模块区域逐一检查对比，确保与设计稿一致，完善遗漏和不准确之处。这是确保分析质量的关键步骤。

**核心检查项（必须执行）**：
1. **区域顺序一致性**：文档中区域排列顺序必须与设计稿从上到下、从左到右顺序一致
2. **布局数据精准性**：位置坐标 (x, y)、尺寸 (宽, 高)、间距 (padding, gap, margin) 必须与设计稿精确匹配
3. **标题文字一致性**：标题文字必须与设计稿完全一致，不得有任何偏差
4. **模块存在性验证**：文档中记录的每个模块必须能在设计稿中找到对应区域，**不得虚构不存在的模块**

## 为什么需要这一步

即使遵循了前面的分析流程，仍可能存在：
- 分析时的遗漏或疏忽
- 对设计稿理解的偏差
- 记录时的笔误或不准确
- 区域边界或嵌套关系的不清晰
- **区域顺序与设计稿不一致**
- **布局数据不够精准**
- **标题文字与设计稿不一致**

通过系统性的区域检查对比，可以：
- 发现并修正遗漏
- 确保分析的准确性
- 提高文档的可执行性
- **确保区域顺序 100% 一致**
- **确保布局数据精准无误**
- **确保标题文字 100% 一致**

## 检查流程

> **⚠️ 检查前必须重新获取设计稿数据**
>
> 所有检查必须基于最新设计稿数据进行：
> - **结构化设计稿（.pen/Figma/MasterGo）**：使用对应 MCP 工具重新获取设计稿结构和节点信息
> - **本地图片**：使用视觉识别能力重新读取图片内容
>
> 禁止仅凭记忆或已提取的缓存数据进行检查。

### 5.1 整体预览检查

**目的**：先从整体角度验证分析结果。

| 检查项 | 说明 |
|--------|------|
| 区域划分 | 所有区域是否都已识别？是否有遗漏区域？ |
| 区域顺序 | 区域列表是否按「从上到下」排序？ |
| 区域边界 | 区域边界是否清晰？是否有重叠或遗漏？ |
| 整体尺寸 | 页面整体尺寸是否正确？ |

**操作方法**：
1. 获取设计稿整体截图或预览
2. 对照已生成的 UI 分析清单
3. 逐区域核对是否有遗漏

### 5.2 区域顺序一致性检查（必须执行）

**目的**：确保文档中区域排列顺序与设计稿完全一致。

**为什么重要**：
- 区域顺序不一致会导致开发实现顺序错误
- 影响前端布局代码的书写顺序
- 可能导致 DOM 结构与设计稿不符

**检查步骤**：

#### 步骤 1：获取设计稿区域顺序

从设计稿中按「从上到下、从左到右」提取所有区域：

**排序规则**：
1. 首先按 y 坐标从小到大排序（从上到下）
2. 当 y 坐标相同时（同一行），按 x 坐标从小到大排序（从左到右）

| 区域名称 | y 坐标（设计稿） | x 坐标（设计稿） | 排序位置 |
|----------|------------------|------------------|----------|
| 区域A | y=0 | x=0 | 第1位 |
| 区域B | y=200 | x=0 | 第2位 |
| 区域C | y=400 | x=132 | 第3位 |
| 区域D | y=400 | x=648 | 第4位（与区域C同行，在右侧）|

#### 步骤 2：对比文档区域顺序

| 文档顺序 | 设计稿顺序 | 是否一致 |
|----------|------------|----------|
| 区域A | 区域A | ✅ |
| 区域B | 区域B | ✅ |
| 区域C | 区域C | ✅ |

#### 步骤 3：发现不一致时的处理

| 问题类型 | 处理方法 |
|----------|----------|
| 区域顺序颠倒 | 重新排列文档中的区域顺序 |
| 区域遗漏 | 补充遗漏的区域分析 |
| 区域重复 | 删除重复的区域记录 |

**区域顺序检查表格模板**：

```markdown
### 区域顺序一致性检查

| 序号 | 设计稿区域（按y排序） | 文档区域 | 状态 |
|------|----------------------|----------|------|
| 1 | 导航栏 (y=0) | 导航栏 | ✅ |
| 2 | 标题区 (y=80) | 标题区 | ✅ |
| 3 | 统计卡片区 (y=160) | 统计卡片区 | ✅ |
| 4 | 列表区 (y=320) | 列表区 | ✅ |
```

**常见顺序问题**：
- 同一行区域未按从左到右顺序记录
- 垂直排列区域顺序颠倒
- 混合布局（上下+左右嵌套）顺序混乱

### 5.3 模块存在性验证（必须执行）

**目的**：确保文档中记录的每个模块都在设计稿中真实存在，防止虚构模块。

**为什么重要**：
- 虚构模块会导致开发实现不需要的功能
- 影响开发工期评估和资源分配
- 验收时产生严重争议
- **破坏文档的可信度**

**常见虚构模块场景**：
| 场景 | 示例 | 问题原因 |
|------|------|----------|
| 根据名称推测添加模块 | 设计稿只有"项目评级情况"，文档额外添加"结算趋势折线图" | 对图表类型误解，未核对设计稿实际内容 |
| 合并相似模块时遗漏 | 设计稿有两个独立图表，文档合并为一个 | 未仔细查看设计稿结构 |
| 参考其他设计稿添加 | 添加了其他页面才有模块 | 混淆了不同设计稿的内容 |
| 预设模板套用错误 | 按固定模板填写，但设计稿结构不同 | 未根据实际设计稿调整分析 |

**检查步骤**：

#### 步骤 1：提取文档中记录的所有模块

从已生成的 UI 分析清单中提取所有模块名称列表。

**示例**：
```markdown
### 文档记录的模块列表

| 序号 | 模块名称 | 章节 |
|------|----------|------|
| 1 | 顶部导航栏 | 2.1 |
| 2 | 页面标题区 | 2.2 |
| 3 | 项目结算总额 | 2.3 |
| 4 | 项目营收平衡情况 | 2.4 |
| 5 | 结算趋势折线图 | 2.5 |
| 6 | 项目评级情况 | 2.6 |
| 7 | 项目情况表格 | 2.7 |
```

#### 步骤 2：在设计稿中逐一验证每个模块

使用设计稿工具，检查每个模块是否真实存在：

**使用 Pencil MCP**：
```javascript
// 获取设计稿整体布局
mcp__pencil__snapshot_layout({
  filePath: "<.pen 文件路径>",
  parentId: "<页面根节点ID>",
  maxDepth: 2
})
```

**使用 Figma MCP**：
```javascript
// 获取 Figma 节点结构
mcp__Framelink_Figma_MCP__get_figma_data({
  fileKey: "<文件Key>",
  nodeId: "<节点ID>"
})
```

#### 步骤 3：创建模块存在性检查表

```markdown
### 模块存在性验证

| 序号 | 文档模块名称 | 设计稿中是否存在 | 验证结果 | 备注 |
|------|--------------|------------------|----------|------|
| 1 | 顶部导航栏 | ✅ 存在 | 通过 | y=0, h=56px |
| 2 | 页面标题区 | ✅ 存在 | 通过 | y=68 |
| 3 | 项目结算总额 | ✅ 存在 | 通过 | x=160, y=112 |
| 4 | 项目营收平衡情况 | ✅ 存在 | 通过 | x=698, y=112 |
| 5 | 结算趋势折线图 | ❌ 不存在 | **虚构模块** | 需删除此模块 |
| 6 | 项目评级情况 | ✅ 存在 | 通过 | x=160, y=552 |
| 7 | 项目情况表格 | ✅ 存在 | 通过 | x=160, y=992 |
```

#### 步骤 4：发现虚构模块时的处理

| 问题类型 | 处理方法 |
|----------|----------|
| 完全虚构的模块 | **立即删除**该模块的整个章节 |
| 模块名称与实际不符 | 更正模块名称为设计稿实际名称 |
| 模块内容归属错误 | 重新划分模块边界，调整内容归属 |

**修正流程**：
1. **删除**虚构模块的整个章节
2. **重新检查**相邻模块的描述是否准确
3. **更新**文档中的模块编号和引用
4. **记录**修正内容到「六、常见遗漏检查点」

#### 步骤 5：修正后的二次验证

修正后，**必须**重新进行模块存在性验证：
- 确认所有虚构模块已删除
- 确认剩余模块均真实存在
- 确认模块编号和章节引用正确

**模块存在性验证清单**：

- [ ] 已提取文档中所有模块名称
- [ ] 已在设计稿中逐一验证每个模块
- [ ] 已创建模块存在性检查表
- [ ] 虚构模块已删除
- [ ] 模块名称已与设计稿核对
- [ ] 修正后已完成二次验证

### 5.4 布局数据精准性检查（必须执行）

**目的**：确保文档中的布局数据与设计稿精确匹配。

**为什么重要**：
- 布局数据不精准会导致还原偏差
- 开发时无法精确还原设计稿
- 验收时产生争议

**检查范围**：

| 数据类型 | 检查项 | 精度要求 |
|----------|--------|----------|
| 位置坐标 | x, y | 精确到像素 |
| 尺寸 | 宽度、高度 | 精确到像素 |
| 内边距 | padding-top/right/bottom/left | 精确到像素 |
| 外边距 | margin-top/right/bottom/left | 精确到像素 |
| 间距 | gap, row-gap, column-gap | 精确到像素 |
| 圆角 | border-radius | 精确到像素 |

**检查步骤**：

#### 步骤 1：从设计稿获取精确布局数据

使用工具获取设计稿中各区域的精确布局数据：

- **Pencil MCP**：使用 `snapshot_layout` 或 `batch_get` 获取节点属性
- **Figma MCP**：使用 `get_figma_data` 获取节点布局信息
- **MasterGo MCP**：使用 `mcp__getDsl` 获取布局数据

#### 步骤 2：逐项对比布局数据

对每个区域创建布局数据对比表：

```markdown
### 区域布局数据检查

| 属性 | 设计稿值 | 文档值 | 状态 |
|------|----------|--------|------|
| x | 132 | 132 | ✅ |
| y | 80 | 80 | ✅ |
| 宽度 | 1176 | 1176 | ✅ |
| 高度 | 240 | 240 | ✅ |
| padding-top | 24 | 24 | ✅ |
| padding-right | 24 | 24 | ✅ |
| padding-bottom | 24 | 24 | ✅ |
| padding-left | 24 | 24 | ✅ |
| gap | 16 | 16 | ✅ |
```

#### 步骤 3：发现数据不精准时的处理

| 问题类型 | 处理方法 |
|----------|----------|
| 数值偏差 | 更正为设计稿精确值 |
| 遗漏属性 | 补充遗漏的布局属性 |
| 方向性错误 | 更正 padding/margin 的方向值 |

**常见数据不精准问题**：
- 间距值四舍五入（如 15.5px 记成 16px）
- padding 四边值不同时只记录了一个值
- gap 方向性遗漏（row-gap 和 column-gap 不同时）
- 相对位置与绝对位置混淆

**布局数据检查清单**：

- [ ] 所有区域的 x, y 坐标已核对
- [ ] 所有区域的宽度、高度已核对
- [ ] 所有区域的 padding 已核对（四边）
- [ ] 所有区域的 margin 已核对
- [ ] 所有区域的 gap 已核对
- [ ] 所有区域的圆角已核对
- [ ] 数据精度达到像素级别

### 5.4 分区域逐一检查

**目的**：对每个区域进行详细检查，确保分析的准确性和完整性。

**按以下顺序进行**：

#### 步骤 1：确定检查顺序

```
从上到下 → 从左到右 → 从外到里
```

与设计稿分析顺序一致，确保检查不遗漏。

#### 步骤 2：逐区域检查清单

对每个区域执行以下检查：

##### A. 文字检查

**⚠️ 优先检查项：标题文字一致性**

| 检查项 | 方法 | 要求 |
|--------|------|------|
| 卡片标题文字 | 逐条对照设计稿 | **必须完全一致**，包括空格、标点、大小写 |
| 区域标题文字 | 逐条对照设计稿 | **必须完全一致**，不得有偏差 |
| 按钮文字 | 逐条对照设计稿 | **必须完全一致** |
| Tab/标签文字 | 逐条对照设计稿 | **必须完全一致** |

**标题文字一致性检查步骤**：
1. 从设计稿中提取该区域的所有标题文字（包括卡片标题、区域标题、按钮文字等）
2. 与已生成的 UI 分析清单逐条对比
3. 发现不一致时，立即修正文档

**标题文字一致性检查示例**：

| 设计稿标题 | 文档记录标题 | 检查结果 |
|------------|--------------|----------|
| 全年结算总额 | 全年结算总额 | ✅ 一致 |
| 部门成员 | 部门成员情况 | ❌ 不一致，需修正 |
| 项目营收平衡情况 | 项目营收平衡 | ❌ 不一致，需修正 |

**常见标题不一致问题**：
- 多字或少字（如「部门成员」记成「部门成员情况」）
- 同义词替换（如「项目列表」记成「项目情况」）
- 简写或缩写（如「近3个月」记成「近三月」）
- 标点符号差异（如「交付部门BI看板」记成「交付部门 BI 看板」）

---

**一般文字检查项**：

| 检查项 | 方法 |
|--------|------|
| 文字内容 | 逐条对照设计稿，确认内容一致 |
| 文字位置 | 确认文字在区域内的位置正确 |
| 文字样式 | 确认字体、字号、字重、颜色、行高 |
| 文字对齐 | 确认对齐方式（左/中/右） |
| 文字数量 | 统计设计稿文字数量与分析记录是否一致 |

**常见问题**：
- 遗漏顶部欢迎语或大标题
- 遗漏按钮/链接文案
- 遗漏占位提示文字
- 遗漏 Tab 或标签文字
- **标题文字与设计稿不一致**

##### B. 图片检查

| 检查项 | 方法 |
|--------|------|
| 图片数量 | 统计设计稿图片数量与分析记录是否一致 |
| 图片尺寸 | 确认每张图片的宽高尺寸 |
| 图片位置 | 确认图片在区域内的位置 |
| 图片比例 | 确认图片的宽高比 |
| 图片圆角 | 确认图片是否有圆角及圆角值 |
| 多图布局 | 确认多图拼接方式和间距 |

**常见问题**：
- 遗漏小图标
- 遗漏占位图
- 多图拼接布局记录不完整
- 图片圆角遗漏

##### C. 布局检查

| 检查项 | 方法 |
|--------|------|
| 区域尺寸 | 确认区域的宽高尺寸 |
| 区域位置 | 确认区域的 x, y 坐标 |
| 内边距 | 确认 padding 值（上右下左） |
| 外边距 | 确认 margin 值 |
| 间距 | 确认 gap 值 |
| 对齐方式 | 确认 align-items, justify-content |
| 排列方向 | 确认 flex-direction 或 grid 布局 |

**常见问题**：
- 间距方向性记录错误
- 对齐方式遗漏或不准确
- 排列方向遗漏

##### D. 层级检查

**⚠️ 强制要求：必须使用 MCP 工具读取实际节点树**

**禁止凭视觉推断层级结构！** 层级必须从设计稿工具返回的实际节点数据中精确提取。

| 检查项 | 方法 |
|--------|------|
| 嵌套关系 | 使用 MCP 工具读取实际 children，确认元素的父子关系 |
| 层级顺序 | 读取实际节点树，确认元素的先后顺序 |
| 层级深度 | 确认是否丢层或嵌套过深 |
| 子节点数量 | 对比设计稿实际子节点数与文档记录数 |

**检查方法**：

1. **使用 MCP 工具读取区域的完整节点树**
   ```python
   # MasterGo 示例
   def extract_hierarchy(node, parent_x=0, parent_y=0, depth=0):
       layout = node.get('layoutStyle', {})
       result = {
           'type': node.get('type', ''),
           'name': node.get('name', ''),
           'children': []
       }
       for child in node.get('children', []):
           result['children'].append(extract_hierarchy(child, ...))
       return result
   ```

2. **将文档层级与实际节点树逐层对比**

3. **核对子节点数量、名称、类型、坐标**

**常见层级推断错误**：

| 错误类型 | 错误推断 | 正确做法 |
|----------|----------|----------|
| 图例位置推断 | "图例应该在标题下面" | 读取实际节点，图例可能在图表区域内右上角 |
| 统计卡片区推断 | "营收平衡是底部统计区" | 读取实际节点，可能是左侧独立卡片组 |
| 图表类型推断 | "这是双轴柱状图" | 读取节点名称，实际可能是堆叠面积图 |
| 子元素虚构 | "应该有筛选条件" | 必须验证设计稿中实际存在 |

**检查表格模板**：

```markdown
### 层级结构检查

| 区域名称 | 设计稿子节点数 | 文档记录子节点数 | 是否一致 | 处理措施 |
|----------|----------------|------------------|----------|----------|
| 项目营收平衡情况 | 4 | 5 | ❌ | 已删除虚构节点 |
| 全年结算总额 | 3 | 3 | ✅ | 无需处理 |
```

**常见问题**：
- 外层容器被忽略
- 嵌套层级丢失
- 顺序颠倒
- **凭视觉推断层级而非读取实际节点**

### 5.5 使用截图对比验证

### 标题一致性专项检查（必须严格执行）

**执行时机**：在每个区域检查完成后，**必须**进行标题一致性专项检查。

**为什么必须执行**：
- 标题文字是 UI 的核心信息，直接影响用户对功能的理解
- 标题不一致会导致开发实现错误，验收时产生争议
- 即使前面步骤已检查，此处仍需**再次核对**

**检查步骤**：

1. **提取设计稿实际标题**
   - 从设计稿中准确复制该区域的所有标题文字
   - 确保提取的是设计稿**实际显示的文字**，而非凭记忆或推测

2. **逐条对比文档记录**
   - 将设计稿实际标题与文档中记录的标题逐条对比
   - 检查是否完全一致（字数、用词、标点、空格）

3. **发现不一致立即修正**
   - 一旦发现不一致，**立即停止后续检查**
   - 修正文档中的标题文字
   - 修正后重新对比确认

**检查表格模板**：

```markdown
### 区域标题一致性检查

| 区域名称 | 设计稿实际标题 | 文档标题 | 状态 | 修正措施 |
|----------|----------------|----------|------|----------|
| 区域1 | 全年结算总额 | 全年结算总额 | ✅ | 无需修正 |
| 区域2 | 部门成员 | 部门成员情况 | ❌ | **已修正**：删除「情况」二字 |
| 区域3 | 项目营收平衡情况 | 项目营收平衡 | ❌ | **已修正**：补充「情况」二字 |
```

**常见标题不一致问题**：
| 问题类型 | 示例（设计稿 → 文档） | 修正方法 |
|----------|----------------------|----------|
| 多字 | 「部门成员」→「部门成员情况」 | 删除多余文字 |
| 少字 | 「项目营收平衡情况」→「项目营收平衡」 | 补充缺失文字 |
| 同义词替换 | 「项目列表」→「项目情况」 | 使用设计稿原词 |
| 简写/缩写 | 「近3个月」→「近三月」 | 使用设计稿原格式 |
| 标点差异 | 「交付部门BI看板」→「交付部门 BI 看板」 | 使用设计稿原格式 |

**修正要求**：
- 发现不一致时，**必须立即修正文档**
- 修正后**必须重新检查确认**
- 在「六、常见遗漏检查点」中记录：
  - 原始错误内容
  - 修正后的内容
  - 修正时间

### 子元素排序专项检查（必须严格执行）

**执行时机**：在标题一致性检查完成后，进行子元素排序专项检查。

**为什么必须执行**：
- **DSL 树结构遍历顺序 ≠ 视觉布局顺序**
- 设计稿工具返回的节点数据按树结构存储，遍历顺序可能与视觉顺序不一致
- 子元素顺序错误会导致开发实现与设计稿不一致

**检查步骤**：

1. **识别横向排列元素组**
   - 找出区域内所有横向排列的子元素组
   - 常见类型：筛选条件区、Tab 标签组、统计卡片组、按钮组、图例项

2. **提取 x 坐标**
   - 使用设计稿工具获取每个子元素的 x 坐标
   - 记录元素名称和 x 坐标值

3. **按 x 坐标排序确认**
   - 将子元素按 x 坐标从小到大排序
   - 确认正确的视觉顺序

4. **对比文档记录**
   - 检查文档中记录的顺序与排序后的顺序是否一致

**检查表格模板**：

```markdown
### 子元素排序检查

| 区域名称 | 子元素组 | 设计稿 x 坐标（排序后） | 文档记录顺序 | 状态 | 处理措施 |
|----------|----------|------------------------|--------------|------|----------|
| 部门项目情况 | 筛选条件 | 项目名称(16) → 项目经理(330) → 交付截止日(645) | 项目经理 → 项目名称 → 交付截止日 | ❌ | 已修正 |
```

**常见排序错误场景**：

| 场景 | 错误原因 | 正确做法 |
|------|----------|----------|
| 按树结构顺序记录 | DSL 返回的 children 数组顺序与视觉顺序不同 | 必须提取 x 坐标后排序 |
| 按名称猜测顺序 | 假设「项目经理」在「项目名称」前面 | 必须核对实际坐标位置 |
| 忽略坐标提取 | 只记录文本内容，不获取位置信息 | 必须获取并记录 x 坐标 |

**修正要求**：
- 发现排序不一致时，**必须立即修正文档**
- 修正后**必须重新检查确认**
- 在「六、常见遗漏检查点」中记录修正内容

### ⚠️ 子元素归属专项检查（必须严格执行）

**执行时机**：在子元素排序检查完成后，进行子元素归属专项检查。

**为什么必须执行**：
- 子元素归属错误是最隐蔽、最难发现的错误类型
- 凭视觉猜测容易将其他区域的子元素错误归属
- 会导致开发实现完全偏离设计稿结构

**检查步骤**：

1. **获取区域实际子元素**
   - 使用 MCP 工具读取区域的实际 children 数据
   - 记录每个子元素的名称和坐标

2. **计算区域边界**
   ```
   父区域范围：
     - 左边界: x = 父区域.x
     - 右边界: x = 父区域.x + 父区域.width
     - 上边界: y = 父区域.y
     - 下边界: y = 父区域.y + 父区域.height
   ```

3. **验证每个子元素归属**
   - 检查子元素的绝对坐标是否在父区域范围内
   - 公式：
     ```
     子元素.x >= 父区域.x
     子元素.x + 子元素.width <= 父区域.x + 父区域.width
     子元素.y >= 父区域.y
     子元素.y + 子元素.height <= 父区域.y + 父区域.height
     ```

4. **对比文档记录**
   - 检查文档中记录的子元素是否都在实际子元素列表中
   - 检查是否有虚构的子元素

**检查表格模板**：

```markdown
### 子元素归属检查

| 区域名称 | 区域边界 (x, y, w, h) | 文档记录子元素 | 子元素实际坐标 | 是否在范围内 | 验证结果 | 处理措施 |
|----------|----------------------|----------------|----------------|--------------|----------|----------|
| 项目结算总额 | 160, 112, 522, 424 | 标题区 | 176, 132 | ✅ 在范围内 | 通过 | 无需处理 |
| 项目结算总额 | 160, 112, 522, 424 | Tab切换区 | 1486, 128 | ❌ x=1486 超出右边界 682 | **归属错误** | 已删除 |
```

**常见归属错误场景**：

| 场景 | 错误示例 | 问题原因 | 正确做法 |
|------|----------|----------|----------|
| Tab 切换归属错误 | 项目结算总额卡片记录了「Tab切换区」，但 Tab 坐标超出卡片范围 | 未验证子元素坐标是否在区域内 | Tab 属于右侧卡片 |
| 图表归属错误 | 左侧卡片记录了「图表区域」，但图表在右侧 | 凭视觉猜测，未使用工具读取 | 需重新划分区域 |
| 子元素虚构 | 文档记录「筛选条件」，但设计稿该区域无筛选条件 | 根据经验推断，未核对设计稿 | 删除虚构子元素 |

**修正要求**：
- 发现归属错误的子元素时，**必须立即修正或删除**
- 发现虚构子元素时，**必须立即删除**
- 修正后**必须重新检查确认**
- 在「六、常见遗漏检查点」中记录修正内容
