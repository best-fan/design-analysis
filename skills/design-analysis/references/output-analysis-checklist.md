### 7.2 主流网页设计常识（实现时需遵循）

详见 `implementation-guidelines.md` 获取详细说明。

**核心原则**：实现时应遵循主流网页设计常识，避免反审美的效果。这些常识应在编写代码时就遵循，而非验收时才发现。

#### 页面容器宽度
- ❌ **不应写死 `max-width`** 导致右侧大片留白（除非设计稿特别说明）
- ✅ **页面应自适应视口宽度**，避免割裂的留白效果

#### 横向列表布局
- ❌ **不应限制一行几个元素而留空白**（如固定列数导致右侧大片空白）
- ❌ **不应出现元素被截断**（如半截卡片显示）
- ✅ **应铺满可用宽度**，能放几个就放几个，不够再换行
- ✅ **换行后应保持列宽一致**（使用 Grid 而非 Flex）

**布局选择指导**：
- **Grid**：适合等列宽、多行一致的情况
  - 使用 `grid-template-columns: repeat(auto-fill, minmax(min, max))` 实现自适应列数
  - 确保 Grid 容器有明确宽度（如 `width: 100%`），否则 `auto-fill` 可能只算出一列
- **Flex**：适合自适应但每行独立计算的情况
  - 需要铺满宽度时使用 `flex-grow: 1`，不要用 `flex: 0 1 fixed-width`
  - 注意 Flex 每行独立计算，换行后列宽可能不一致

#### 元素展示完整性
- ❌ **不应展示不完整的元素**（如半截卡片、被裁切的按钮）
- ✅ **要么完整展示，要么完全不展示**（使用 `overflow: hidden` 配合数量控制，而非视觉截断）

#### 对齐方式
- ✅ **图标与文本的对齐**：检查 `align-items` 值，常见应为 `center`（水平居中）而非 `flex-start`（顶部对齐）
- ✅ **按钮、卡片等元素的定位**：仔细对照设计稿的 `padding`、`margin`、定位规则

#### CSS 优先级与样式覆盖
- ⚠️ **组件样式被页面级样式覆盖**：传入的 `className` 可能覆盖组件内部的关键样式（如 `display: block` 覆盖 `display: flex`）
- ✅ **建议**：组件内部样式使用更具体的选择器，或使用 CSS Modules 避免冲突
- ✅ **检查**：验收时检查组件样式是否被父级样式覆盖

#### 必须用浏览器验证的细节
- ⚠️ **按钮高度、宽度**：必须用浏览器工具实际查看渲染效果，不能仅凭代码推测
- ⚠️ **对齐效果**：必须用浏览器查看实际对齐效果
- ⚠️ **hover/active 状态**：必须用浏览器实际触发查看
- ⚠️ **响应式表现**：必须用浏览器在不同视口下查看

**看到设计稿时，默认不会出现以下反审美的效果**：
- 右侧大片留白
- 列表留空白
- 元素截断
- 换行后列宽不一致

### 7.3 常见错误模式与避免方法

详见 `implementation-common-errors.md` 获取详细说明。

| 错误模式 | 原因 | 避免方法 |
|---------|------|----------|
| **Grid 一行只有一个** | Grid 容器宽度不确定，`auto-fill` 只算出一列 | 确保 Grid 容器有明确宽度（如 `width: 100%`） |
| **Flex 不铺满、留空白** | 使用了 `flex: 0 1 fixed-width`，不允许增长 | 使用 `flex-grow: 1` 或 `flex: 1 1 min-width` |
| **换行后列宽不一致** | Flex 布局每行独立计算 | 使用 Grid 而非 Flex 来保证列宽一致 |
| **组件样式被覆盖** | 页面级样式覆盖了组件内部样式 | 检查 CSS 优先级，使用更具体的选择器或 CSS Modules |
| **对齐方式错误** | 默认使用了 `flex-start` 而非 `center` | 明确设置 `align-items: center` |
| **按钮尺寸不对** | 未用浏览器实际查看，仅凭代码推测 | **必须用浏览器工具查看实际渲染效果** |

## 生成后校验（必须执行）

UI 分析清单文档生成完成后，必须执行以下校验步骤：

> **⚠️ 校验前必须重新获取设计稿数据**
>
> 所有校验必须基于最新设计稿数据进行：
> - **结构化设计稿（.pen/Figma/MasterGo）**：使用对应 MCP 工具重新获取设计稿结构和节点信息
> - **本地图片**：使用视觉识别能力重新读取图片内容
>
> 禁止仅凭记忆或已提取的缓存数据进行校验。

### ⚠️ 使用分析脚本避免编码问题

**推荐使用脚本进行校验**，避免控制台编码问题：

```bash
# MasterGo 设计稿校验
# 1. 区域数量校验
python scripts/mastergo-dsl-analyzer.py <dsl_file>

# 2. 标题提取（用于校验三）
python scripts/mastergo-dsl-analyzer.py <dsl_file> --titles

# 标题会保存到 openspec/output/card_titles.json
# 然后使用 Read 工具读取该文件进行对比
```

**优势**：
- 脚本输出到文件，使用 UTF-8 编码，避免控制台乱码
- 结果可持久保存，便于多次校验
- 格式标准化，便于对比

### ⚠️ 校验零：区域数量校验（必须首先执行）

**目的**：确保文档记录的区域数量与设计稿实际区域数量完全一致，**防止遗漏任何区域**。

**重要性**：
- 区域遗漏是最常见、最严重的错误类型
- 遗漏底部区域会导致开发实现不完整
- 影响工期评估和验收标准

**强制步骤**：

1. **统计设计稿顶层节点数量**
   - 使用设计稿工具获取根节点的所有直接子节点
   - 过滤出 GROUP/FRAME/INSTANCE 类型且有实际尺寸（w > 100, h > 50）的节点
   - 记录总数
   - **推荐使用分析脚本**：
     - MasterGo：`python scripts/mastergo-dsl-analyzer.py <dsl_file>`
     - 脚本会自动输出"主要区域数量"，避免手动统计遗漏

2. **统计文档区域列表数量**
   - 计算「一、设计稿整体分析」中「区域列表」表格的行数

3. **数量对比**
   - 设计稿实际区域数 vs 文档记录区域数
   - 若不一致，**必须立即查找遗漏区域**

**校验表格**：

```markdown
### 区域数量校验

| 项目 | 数量 | 是否一致 |
|------|------|----------|
| 设计稿顶层区域数 | ___ 个 | - |
| 文档区域列表行数 | ___ 行 | ✅ / ❌ |

**若不一致，遗漏区域**：
- [ ] 已逐个对比，找出遗漏区域
- [ ] 已补充遗漏区域到文档
- [ ] 已重新执行数量校验
```

**常见遗漏场景**：
| 场景 | 问题原因 | 检查方法 |
|------|----------|----------|
| 只分析首屏 | 忽略 y > 1080 的区域 | 检查所有 y 坐标 |
| 依赖 Agent 结果 | Agent 可能遗漏 | 亲自遍历节点 |
| 疏忽底部表格/图表 | 关注上方卡片 | 按页面高度完整扫描 |
| **宽区域遗漏** | 区域宽度接近页面宽度，被误判为"容器"过滤 | 使用官方脚本，不过滤接近页面宽度的区域 |

**修正要求**：
- **发现区域遗漏时，必须立即补充**
- 补充完整的区域分析（文字、图片、布局、层级）
- 重新执行区域数量校验

### 校验一：区域顺序校验

**目的**：确保「一、设计稿整体分析」中的「区域列表」按「从上到下、从左到右」排序。

**校验步骤**：

1. **获取设计稿区域坐标**
   - 从设计稿中提取所有区域的 x、y 坐标
   - 记录每个区域的名称和位置

2. **计算正确排序**
   - 首先按 y 坐标从小到大排序（从上到下）
   - 当 y 坐标相同时（同一行），按 x 坐标从小到大排序（从左到右）

3. **对比验证**
   - 将文档中的区域顺序与计算出的正确排序对比
   - 标记不一致的区域

**校验表格**：

```markdown
### 区域顺序校验

| 序号 | 区域名称 | 设计稿 y | 设计稿 x | 文档顺序 | 是否正确 | 备注           |
| ---- | -------- | -------- | -------- | -------- | -------- | -------------- |
| 1    | 导航栏   | 0        | 0        | 第1位    | ✅       | 顶部固定区域   |
| 2    | 标题区   | 80       | 0        | 第2位    | ✅       | 导航栏下方     |
| 3    | 左侧边栏 | 160      | 0        | 第3位    | ✅       | 与右侧内容同行 |
| 4    | 右侧内容 | 160      | 240      | 第4位    | ✅       | 在左侧边栏右侧 |
```

**修正要求**：

- 若发现顺序不一致，立即调整文档中区域的排列顺序
- 调整后在「六、常见遗漏检查点」中记录修正内容

### 校验二：区域详细分析顺序校验

**目的**：确保「二、区域详细分析」中的区域顺序与「区域列表」一致。

**校验步骤**：

1. 获取「区域列表」中的区域顺序
2. 获取「区域详细分析」中的区域顺序（按 ### 区域X：名称 的顺序）
3. 对比两者是否完全一致

**校验示例**：

| ------------ | ---------------- | ----------------- |

| 2. 标题区    | 2. 标题区        | ✅                |

### 校验三：标题文字一致性校验（必须严格执行）

**目的**：确保各区域的标题文字与设计稿实际内容完全一致，不得有任何偏差。

**重要性**：

- 标题文字是 UI 的核心信息，直接影响用户对功能的理解
- 标题不一致会导致开发实现错误，验收时产生争议
- 文字错误（多字、少字、错别字）会降低产品专业度

**⚠️ 强制要求：使用文件输出避免编码问题**

**问题背景**：Windows 控制台输出中文时可能遇到编码问题（GBK 编码无法处理某些字符），导致输出乱码。**遇到乱码时，禁止假设校验通过！**

**正确做法**：
```python
# ❌ 错误：直接打印到控制台（可能乱码）
print(f'标题: {title}')

# ✅ 正确：保存到文件，使用 UTF-8 编码
with open('openspec/output/card_titles.json', 'w', encoding='utf-8') as f:
    json.dump(titles, f, ensure_ascii=False, indent=2)

# 然后使用 Read 工具读取文件内容进行对比
```

**校验步骤**：

1. **提取设计稿标题（必须使用文件输出）**
   - 使用设计稿工具提取所有卡片/区域标题
   - **强制保存到 JSON 文件**（使用 `ensure_ascii=False` 参数）
   - 使用 Read 工具读取文件确认实际文字
   - **禁止根据图表类型或经验推测标题**

2. **逐条对比检查（必须使用校验表格）**
   - 使用以下表格格式记录对比结果
   - 每个标题必须明确标记「✅ 一致」或「❌ 不一致」
   - 不允许跳过任何标题

3. **标记不一致项**
   - 发现不一致时，标记具体位置和差异内容
   - 记录设计稿实际文字 vs 文档记录文字
   - **立即修正文档**

**重点检查类型**：

2. **计算区域边界**
   ```
   父区域范围：
     - 左边界: x = 父区域.x
     - 右边界: x = 父区域.x + 父区域.width
     - 上边界: y = 父区域.y
     - 下边界: y = 父区域.y + 父区域.height
   ```

3. **验证每个子元素归属**
   - 检查子元素的绝对坐标是否在父区域范围内
   - 公式：
     ```
     子元素.x >= 父区域.x
     子元素.x + 子元素.width <= 父区域.x + 父区域.width
     子元素.y >= 父区域.y
     子元素.y + 子元素.height <= 父区域.y + 父区域.height
     ```

4. **对比文档记录**
   - 检查文档中记录的子元素是否都在实际子元素列表中
   - 检查是否有虚构的子元素

**校验表格模板**：

```markdown
### 子元素归属校验

| 区域名称 | 区域边界 (x, y, w, h) | 文档记录子元素 | 子元素实际坐标 | 是否在范围内 | 验证结果 | 处理措施 |
|----------|----------------------|----------------|----------------|--------------|----------|----------|
| 项目结算总额 | 160, 112, 522, 424 | 标题区 | 176, 132 | ✅ 在范围内 | 通过 | 无需处理 |
| 项目结算总额 | 160, 112, 522, 424 | 数值展示区 | 327, 275 | ✅ 在范围内 | 通过 | 无需处理 |
| 项目结算总额 | 160, 112, 522, 424 | 年份选择器 | 546, 128 | ✅ 在范围内 | 通过 | 无需处理 |
| 项目结算总额 | 160, 112, 522, 424 | Tab切换区 | 1486, 128 | ❌ x=1486 超出右边界 682 | **归属错误** | Tab不属于此卡片，已删除 |
| 项目结算总额 | 160, 112, 522, 424 | 图表区域 | 未在设计稿中找到 | ❌ 不存在 | **虚构子元素** | 已删除 |
```

**修正要求**：

- **发现归属错误的子元素时**：
  - 如果子元素属于其他区域，修正归属关系
  - 如果子元素不存在于任何区域（设计稿外的元素），删除该记录
- **发现虚构子元素时**：
  - **必须立即删除**该子元素的记录
- 在「六、常见遗漏检查点」中记录修正内容
- 修正后重新执行校验，确认所有子元素归属正确

**验证示例**：

```
设计稿数据：
  项目结算总额卡片 (M7UEE): x=160, y=112, width=522, height=424
  → 右边界 = 160 + 522 = 682

  Tab切换: x=1486, y=128

验证：
  Tab切换.x (1486) > 卡片右边界 (682)
  → Tab切换不属于项目结算总额卡片
  → 属于项目营收平衡情况卡片 (x=698, width=1062, 右边界=1760)
```

### ⚠️ 校验十：层级结构校验（必须严格执行）

**目的**：确保文档中记录的层级结构与设计稿实际节点树完全一致，防止凭视觉推断导致的错误。

**重要性**：

- 层级结构是"四类重中之重"之一
- 凭视觉推断层级会导致严重错误（如图例位置、卡片归属、图表类型）
- 影响开发实现的结构准确性
- 验收时产生争议

**常见层级推断错误**：

| 错误类型 | 错误推断 | 正确做法 |
|----------|----------|----------|
| 图例位置推断 | "图例应该在标题下面" | 读取实际节点，图例可能在图表区域内右上角 |
| 统计卡片区推断 | "营收平衡是底部统计区" | 读取实际节点，可能是左侧独立卡片组 |
| 图表类型推断 | "这是双轴柱状图" | 读取节点名称，实际可能是堆叠面积图 |
| 子元素归属推断 | "Tab 属于这个卡片" | 验证 Tab 坐标是否在卡片范围内 |

**校验步骤**：

1. **逐区域提取实际节点树**
   - 使用 MCP 工具获取每个区域的完整 children 数据
   - 递归提取所有子节点的类型、名称、坐标、尺寸

2. **构建实际层级结构**
   ```
   区域
   ├── [TYPE] 子节点名称 (x, y, wxh)
   │   ├── [TYPE] 孙节点名称
   │   └── [TYPE] 孙节点名称
   └── [TYPE] 子节点名称
   ```

3. **逐层对比校验**
   - 子节点数量是否一致
   - 子节点名称/类型是否一致
   - 子节点坐标是否准确
   - 嵌套关系是否正确

**校验表格模板**：

```markdown
### 层级结构校验

| 区域名称 | 设计稿子节点数 | 文档记录子节点数 | 是否一致 | 处理措施 |
|----------|----------------|------------------|----------|----------|
| 项目营收平衡情况 | 4 (标题区、Tab区、左侧卡片组、图表区) | 4 | ✅ | 无需处理 |
| 全年结算总额 | 3 (标题区、数值区、结算分类区) | 4 | ❌ | 已删除虚构节点 |
```

**Python 示例代码**：

```python
def verify_hierarchy(doc_hierarchy, dsl_node, parent_x=0, parent_y=0):
    """校验层级结构"""
    errors = []

    layout = dsl_node.get('layoutStyle', {})
    abs_x = parent_x + (layout.get('relativeX') or 0)
    abs_y = parent_y + (layout.get('relativeY') or 0)

    # 获取实际子节点
    actual_children = dsl_node.get('children', [])

    # 对比数量
    if len(doc_hierarchy.get('children', [])) != len(actual_children):
        errors.append(f"子节点数量不一致: 文档 {len(doc_hierarchy.get('children', []))}, 设计稿 {len(actual_children)}")

    # 递归校验
    for i, actual_child in enumerate(actual_children):
        if i < len(doc_hierarchy.get('children', [])):
            child_errors = verify_hierarchy(
                doc_hierarchy['children'][i],
                actual_child,
                abs_x, abs_y
            )
            errors.extend(child_errors)

    return errors
```

**修正要求**：

- **发现层级不一致时，必须立即修正文档**
- 使用 MCP 工具重新读取实际节点树
- 按实际节点树重新记录层级结构
- 修正后重新执行校验，确认完全一致
- 在「六、常见遗漏检查点」中记录修正内容
