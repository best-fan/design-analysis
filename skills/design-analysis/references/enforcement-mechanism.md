# 强制执行机制

## 问题背景

**规范写得再清楚，如果执行者没有建立强制执行机制，就会被绕过。**

### 常见问题

| 问题类型 | 表现 | 后果 |
|----------|------|------|
| 读了规范但不执行 | 「我知道了」→ 开始工作 → 跳过关键步骤 | 输出错误文档 |
| 凭经验推断 | 「柱状图很常见，应该是柱状图」 | 图表类型错误 |
| 过度依赖脚本 | 脚本只输出基本信息，不覆盖图表类型、层级结构 | 遗漏关键信息 |
| 跳过校验步骤 | 「看起来没问题了」→ 跳过校验一到十 | 错误未被发现 |
| **脚本输出截断** | **脚本只输出前N条数据，后续数据未显示** | **遗漏元素** |
| **未读取完整数据** | **依赖控制台输出，未读取保存的完整JSON** | **遗漏元素** |

### 根本原因

```
❌ 当前路径：读取规范 → 理解规范 → 开始工作 → 凭经验/走捷径 → 输出（可能错误）
✅ 正确路径：读取规范 → 理解规范 → 强制检查清单 → 逐项打勾 → 输出 → 逐项校验 → 完成
```

### ⚠️ 真实遗漏案例（2026-03-23）

**案例：项目营收平衡情况卡片遗漏「营收平衡」数值组**

| 步骤 | 问题 | 后果 |
|------|------|------|
| 脚本输出截断 | 脚本设置 `texts_sorted[:15]`，只显示前15条 | 后15条未显示 |
| 依赖控制台输出 | 未读取 `region_details.json` 完整内容 | 看不到完整数据 |
| 认知偏差 | 假设只有「结算收入」「成本支出」2个数值组 | 未验证是否有第3组 |
| 校验缺失 | 无结构化校验流程 | 遗漏未被发现 |

**遗漏内容**：
- 左侧数值3「+165.8 万」
- 左侧标签3「营收平衡」

## 强制执行机制

### 机制一：分析前强制检查清单

**在开始分析任何区域之前，必须完成以下检查项并逐项打勾：**

```markdown
## 分析前强制检查清单

### 基础信息获取
- [ ] 已获取设计稿整体尺寸
- [ ] 已统计顶层区域数量
- [ ] 已按 y,x 坐标排序区域列表

### 工具准备
- [ ] 已确认设计稿类型（.pen / Figma / MasterGo / 本地图片）
- [ ] 已确认可用的 MCP 工具
- [ ] 已准备好分析脚本（如适用）

### 禁止事项确认
- [ ] **我承诺：不凭经验推断图表类型**
- [ ] **我承诺：不凭经验推断子元素归属**
- [ ] **我承诺：每个区域都用 MCP 工具读取实际子节点**
```

### 机制二：区域分析强制检查清单

**在分析每个区域之前，必须完成以下检查项：**

```markdown
## 区域分析强制检查清单（每个区域必须执行）

### 步骤 1：读取实际子节点（必须执行）
- [ ] **已使用 MCP 工具读取该区域的实际 children 数据**
- [ ] **已记录子节点数量**
- [ ] **已记录每个子节点的 name 和 type**

### 步骤 2：读取完整数据文件（⚠️ 新增，必须执行）
- [ ] **已保存完整数据到 JSON 文件**
- [ ] **已使用 Read 工具读取完整的 JSON 文件内容**
- [ ] **禁止仅依赖控制台输出（控制台可能截断）**
- [ ] **已统计文字节点总数：___ 个**
- [ ] **已确认读取了所有文字节点（无截断）**

### 步骤 3：识别图表类型（必须执行）
- [ ] **已读取图表节点的 name 字段**
- [ ] **已确认实际图表类型**（禁止推断）

常见图表类型关键词：
| 节点 name 中的关键词 | 实际图表类型 |
|---------------------|-------------|
| `Stacked area chart` | 堆叠面积图 |
| `堆叠柱状图` | 堆叠柱状图 |
| `分组柱状图` | 分组柱状图 |
| `Bar` | 柱状图 |
| `Line` | 折线图 |
| `Pie` | 饼图 |

### 步骤 4：记录层级结构（必须执行）
- [ ] **已使用 MCP 工具读取节点的完整 children 结构**
- [ ] **已按实际节点树记录层级**（禁止凭视觉推断）

### 步骤 5：子元素归属验证（必须执行）
- [ ] **已计算该区域的边界范围**（x, y, width, height）
- [ ] **已验证每个子元素坐标是否在范围内**
- [ ] **已删除归属错误的子元素记录**

### 步骤 6：结构化元素统计（⚠️ 新增，必须执行）
**这是防止遗漏的关键步骤，必须逐项统计并验证数量**

```markdown
## 结构化元素统计清单

### 数值组统计（针对含数值的卡片）
- [ ] **已统计数值组总数：___ 组**
- [ ] 第1组：标签「___」，数值「___」
- [ ] 第2组：标签「___」，数值「___」
- [ ] 第3组：标签「___」，数值「___」← 必须验证是否有第3组
- [ ] 第4组：标签「___」，数值「___」← 必须验证是否有第4组
- [ ] **确认无更多数值组**

### Tab 切换统计
- [ ] **已统计 Tab 总数：___ 个**
- [ ] Tab1：「___」（选中/未选中）
- [ ] Tab2：「___」
- [ ] Tab3：「___」
- [ ] **确认无更多 Tab**

### 图例项统计
- [ ] **已统计图例项总数：___ 个**
- [ ] 图例1：「___」
- [ ] 图例2：「___」
- [ ] **确认无更多图例项**
```

### 步骤 7：常见组件检查（必须执行，防止遗漏）

**⚠️ 这是本次新增的强制检查项，用于防止遗漏嵌套在深层节点中的常见组件。**

常见组件是极易被遗漏的元素，因为它们通常嵌套在 INSTANCE 组件内部，深度遍历时容易跳过。

```markdown
## 常见组件检查清单（每个区域必须执行）

### Tab 切换检查
- [ ] **已使用工具搜索「tab」「页签」「标签页」「切换」关键词**
- [ ] Tab 组件存在：是 / 否
- [ ] 若存在，Tab 数量：___ 个
- [ ] Tab 标签文字：___

### 图例检查
- [ ] **已使用工具搜索「图例」「legend」关键词**
- [ ] 图例存在：是 / 否
- [ ] 若存在，图例项：___

### 分页检查
- [ ] **已使用工具搜索「分页」「pagination」「页」关键词**
- [ ] 分页组件存在：是 / 否

### 筛选条件检查
- [ ] **已使用工具搜索「筛选」「filter」「搜索」「选择器」关键词**
- [ ] 筛选条件存在：是 / 否
- [ ] 若存在，筛选条件数量：___ 个
```

#### 常见组件遗漏案例

| 遗漏组件 | 遗漏原因 | 正确做法 |
|----------|----------|----------|
| Tab 切换 | 嵌套在 INSTANCE 组件内部，深度遍历默认只到 3 层 | 使用关键词搜索，增加遍历深度到 6 层 |
| 图例 | 节点名称可能不包含「图例」字样 | 同时搜索「legend」和检查图表区域内的文字节点 |
| 分页 | 分页通常在表格区域底部，容易被忽略 | 在分析表格区域时，强制检查是否有分页组件 |
| **数值组** | **脚本输出截断，只显示前N条** | **读取完整JSON文件，统计所有数值组** |
```

### 机制三：输出前强制检查清单

**在输出文档之前，必须完成以下检查项：**

```markdown
## 输出前强制检查清单

### 禁止事项最终确认
- [ ] **我确认：没有凭经验推断任何图表类型**
- [ ] **我确认：每个区域都用 MCP 工具读取了实际子节点**
- [ ] **我确认：没有虚构任何子元素或模块**

### 文档完整性检查
- [ ] 包含所有 9 个章节
- [ ] 区域列表与设计稿区域数量一致
- [ ] 每个区域的层级结构与设计稿一致
```

### 机制四：校验强制执行

**输出文档后，必须逐项执行校验零到校验十，不可跳过任何一项：**

```markdown
## 校验强制执行清单

### 校验零：区域数量校验（必须首先执行）
- [ ] 设计稿顶层区域数 = ___ 个
- [ ] 文档区域列表行数 = ___ 行
- [ ] 数量一致：✅ / ❌
- [ ] 若不一致，已补充遗漏区域

### 校验一：区域顺序校验
- [ ] 已按 y,x 坐标排序
- [ ] 文档顺序与设计稿一致

### 校验二：区域详细分析顺序校验
- [ ] 区域详细分析顺序与区域列表一致

### 校验三：标题文字一致性校验
- [ ] 已逐条对比标题文字
- [ ] 所有标题完全一致

### 校验四：模块存在性验证
- [ ] 已在设计稿中验证每个模块存在
- [ ] 无虚构模块

### 校验五：区域列表一致性校验
- [ ] 已逐区域核对设计稿实际内容
- [ ] 区域标题完全一致

### 校验六：表格列标题专项校验
- [ ] 已逐列读取表格列标题
- [ ] 列标题与设计稿完全一致

### 校验七：区域列表包含子元素一致性校验
- [ ] 已逐区域提取设计稿实际子元素
- [ ] 子元素描述准确

### 校验八：子元素排序校验
- [ ] 已按 x 坐标排序横向子元素
- [ ] 文档记录顺序正确

### 校验九：子元素归属校验
- [ ] 已用 MCP 工具读取实际子元素
- [ ] 所有子元素归属正确
- [ ] 无虚构子元素

### 校验十：层级结构校验
- [ ] 已用 MCP 工具提取实际节点树
- [ ] 层级结构与设计稿一致
- [ ] 未凭视觉推断层级
```

## 禁止行为清单

### 禁止凭经验推断

| 禁止行为 | 正确做法 |
|----------|----------|
| ❌ 「营收对比通常用柱状图」→ 记录为柱状图 | ✅ 读取节点 name 字段 → 发现 "Stacked area chart" → 记录为堆叠面积图 |
| ❌ 「评级对比通常用分组柱状图」→ 记录为分组柱状图 | ✅ 读取节点 name 字段 → 发现 "堆叠柱状图" → 记录为堆叠柱状图 |
| ❌ 「这个区域应该有图表」→ 假设有图表 | ✅ 读取实际 children → 发现无图表节点 → 记录为无图表 |

### 禁止跳过步骤

| 禁止行为 | 后果 | 正确做法 |
|----------|------|----------|
| ❌ 跳过「用 MCP 工具读取子节点」 | 子元素归属错误 | ✅ 每个区域都必须读取 |
| ❌ 跳过「读取图表节点 name 字段」 | 图表类型错误 | ✅ 每个图表都必须读取 |
| ❌ 跳过校验一到校验十 | 错误未被发现 | ✅ 逐项执行校验 |

### 禁止虚构内容

| 禁止行为 | 后果 | 正确做法 |
|----------|------|----------|
| ❌ 文档记录了设计稿中不存在的模块 | 开发实现不需要的功能 | ✅ 在设计稿中验证每个模块存在 |
| ❌ 文档记录了设计稿中不存在的子元素 | 开发实现错误的结构 | ✅ 用 MCP 工具验证每个子元素存在 |

## 检查清单模板

### 分析过程中使用

在分析每个区域时，复制以下模板到对话中，逐项打勾：

```markdown
## 区域 [区域名称] 分析检查

### 步骤 1：读取实际子节点
- [ ] 已使用 MCP 工具读取 children 数据
- [ ] 子节点数量：___ 个
- [ ] 子节点列表：___

### 步骤 2：识别图表类型
- [ ] 已读取图表节点 name 字段
- [ ] 图表类型：___（禁止推断）

### 步骤 3：记录层级结构
- [ ] 已读取完整 children 结构
- [ ] 层级结构：
  ```
  区域
  ├── 子节点1
  ├── 子节点2
  └── ...
  ```

### 步骤 4：子元素归属验证
- [ ] 区域边界：(x=___, y=___, w=___, h=___)
- [ ] 所有子元素坐标在范围内
```
