# 第二步：区域与元素提取

## 概述

对**每个区域**按「从外到里」逐项提取，避免只做「大致印象」式记录。**文字、图片、布局、层级四者均为重中之重，必须逐一准确记录，不允许出错或遗漏。**

详见 `analysis-priorities.md`。

## 2.1 子元素提取强制规则（必须遵守）

### ⚠️ 强制前提：必须使用 MCP 工具读取实际子元素

**禁止凭视觉猜测或凭经验推断子元素归属！**

在提取任何区域的子元素之前，必须使用 MCP 工具读取该区域的**实际子节点数据**：

**使用 Pencil MCP**：
```javascript
// 获取区域的实际子元素
mcp__pencil__batch_get({
  filePath: "<.pen 文件路径>",
  nodeIds: ["<区域节点ID>"],
  readDepth: 3  // 读取到第3层，获取子元素详情
})
```

**使用 Figma MCP**：
```javascript
mcp__Framelink_Figma_MCP__get_figma_data({
  fileKey: "<文件Key>",
  nodeId: "<节点ID>"
})
```

**使用 MasterGo MCP**：
```javascript
mcp__mastergo-magic-mcp__mcp__getDsl({
  fileId: "<文件ID>",
  layerId: "<图层ID>"
})
```

### ⚠️ 强制要求：必须读取完整数据文件（新增）

**禁止仅依赖控制台输出！控制台可能截断数据！**

**问题背景**：
- 分析脚本通常有输出限制（如只显示前15条）
- 控制台输出不完整会导致遗漏
- 必须读取保存的完整 JSON 文件

**强制步骤**：

1. **运行分析脚本，保存数据到 JSON 文件**
   ```bash
   python scripts/analyze-mastergo-dsl.py
   # 数据保存到 openspec/output/region_details.json
   ```

2. **使用 Read 工具读取完整的 JSON 文件**
   ```
   Read("openspec/output/region_details.json")
   ```

3. **统计元素数量，确认无遗漏**
   - 文字节点总数：___ 个
   - 数值组总数：___ 组
   - Tab 总数：___ 个

4. **逐项核对，确保所有元素都已记录**

**错误示例**：
```
❌ 错误：只看脚本控制台输出
  脚本输出：texts_sorted[:15]  # 只显示前15条
  结果：遗漏后15条文字

✅ 正确：读取完整 JSON 文件
  Read("openspec/output/region_details.json")
  统计：文字节点总数 30 个
  确认：已记录所有 30 个文字节点
```

### ⚠️ 子元素归属验证（必须执行）

**验证规则**：子元素的绝对坐标必须在父区域的坐标范围内。

```
父区域范围：
  - 左边界: x = 父区域.x
  - 右边界: x = 父区域.x + 父区域.width
  - 上边界: y = 父区域.y
  - 下边界: y = 父区域.y + 父区域.height

子元素必须在范围内：
  - 子元素.x >= 父区域.x
  - 子元素.x + 子元素.width <= 父区域.x + 父区域.width
  - 子元素.y >= 父区域.y
  - 子元素.y + 子元素.height <= 父区域.y + 父区域.height
```

**常见归属错误**：
| 错误类型 | 示例 | 正确归属 |
|----------|------|----------|
| Tab 切换归属错误 | 项目结算总额卡片记录了 Tab 切换，但 Tab 坐标 (x=1486) 超出卡片范围 (x=160~682) | Tab 属于右侧卡片 |
| 图表归属错误 | 左侧卡片记录了图表，但图表在右侧 | 需重新划分区域 |
| 子元素越界 | 子元素 x=700 被记录在 x=160, width=500 的卡片内 | 子元素不属于该卡片 |

### ⚠️ 核心原则：按坐标排序，而非树结构顺序

**DSL 树结构遍历顺序 ≠ 视觉布局顺序**

设计稿工具返回的节点数据是按树结构存储的，遍历顺序可能与视觉上的从左到右顺序不一致。**必须按坐标重新排序后记录。**

### 强制排序规则

**所有横向排列的子元素，必须按 x 坐标从小到大排序后记录。**

```
错误示例（按树结构顺序）：
  筛选条件：项目经理 → 项目名称 → 交付截止日

正确示例（按 x 坐标排序）：
  筛选条件：项目名称 (x=16) → 项目经理 (x=330) → 交付截止日 (x=645)
```

### 排序方法

1. **获取区域内所有子元素**
2. **提取每个子元素的 x、y 坐标**
3. **按 坐标排序**：先按 y 排序，y 相同时按 x 排序
4. **按排序后的顺序记录**

## 2.2 图表类型识别强制规则（必须遵守）

### ⚠️ 强制要求：必须读取节点 name 字段确认图表类型

**禁止凭经验推断图表类型！**

图表类型是高频错误区，必须从设计稿工具返回的实际节点数据中精确提取。

### 禁止行为

| 禁止行为 | 错误示例 | 正确做法 |
|----------|----------|----------|
| 凭页面名称推断 | 「营收平衡」→ 推断是柱状图 | 读取节点 name → 发现 "Stacked area chart" → 堆叠面积图 |
| 凭经验推断 | 「评级对比通常用分组柱状图」 | 读取节点 name → 发现 "堆叠柱状图" → 堆叠柱状图 |
| 假设有图表 | 「这个卡片应该有图表」 | 读取实际 children → 发现无图表节点 → 记录为无图表 |

### 常见图表类型关键词

| 节点 name 中的关键词 | 实际图表类型 |
|---------------------|-------------|
| `Stacked area chart` / `堆叠面积图` | 堆叠面积图 |
| `堆叠柱状图` | 堆叠柱状图 |
| `分组柱状图` | 分组柱状图 |
| `Bar` / `柱状图` | 柱状图 |
| `Line` / `折线图` | 折线图 |
| `Pie` / `饼图` | 饼图 |
| `Radar` / `雷达图` | 雷达图 |
| `Scatter` / `散点图` | 散点图 |

### 图表类型识别步骤（必须执行）

1. **使用 MCP 工具读取区域的 children 数据**
2. **搜索包含图表关键词的节点**（chart、图表、Bar、Line、Area 等）
3. **读取该节点的 name 字段**
4. **根据 name 字段确认实际图表类型**
5. **记录到文档中**

### 错误案例

**案例：项目营收平衡情况卡片**

```
❌ 错误做法：
  看到「营收平衡」→ 推断是柱状图 → 记录为「柱状图」

✅ 正确做法：
  使用 MCP 工具读取 children → 发现节点 name="Stacked area chart堆叠"
  → 确认是「堆叠面积图」→ 记录为「堆叠面积图」
```

**案例：项目评级情况卡片**

```
❌ 错误做法：
  看到「评级对比」→ 推断是分组柱状图 → 记录为「分组柱状图」

✅ 正确做法：
  使用 MCP 工具读取 children → 发现节点 name="图例/堆叠柱状图"
  → 确认是「堆叠柱状图」→ 记录为「堆叠柱状图」
```

```python
def get_sorted_children(node):
    """获取按坐标排序的子元素"""
    children = node.get('children', [])

    # 提取坐标信息
    children_with_coords = []
    for child in children:
        layout = child.get('layoutStyle', {})
        children_with_coords.append({
            'name': child.get('name'),
            'x': layout.get('relativeX', 0),
            'y': layout.get('relativeY', 0),
            'node': child
        })

    # 按 y 排序，y 相同时按 x 排序
    children_with_coords.sort(key=lambda c: (c['y'], c['x']))

    return children_with_coords
```

### 必须遵守的场景

| 场景 | 说明 |
|------|------|
| 筛选条件区 | 多个筛选输入框，必须按 x 坐标排序 |
| Tab 标签组 | 多个 Tab 标签，必须按 x 坐标排序 |
| 统计卡片组 | 多个横向卡片，必须按 x 坐标排序 |
| 按钮组 | 多个横向按钮，必须按 x 坐标排序 |
| 表格列 | 表格列标题，必须按 x 坐标排序 |
| 图例项 | 图表图例项，必须按 x 坐标排序 |
| 表单字段组 | 横向排列的表单字段，必须按 x 坐标排序 |

### 常见错误警示

| 错误类型 | 错误示例 | 正确做法 |
|----------|----------|----------|
| 按树结构顺序记录 | 直接遍历 children 数组记录 | 先提取坐标，排序后记录 |
| 按名称猜测顺序 | 假设「项目经理」在「项目名称」前面 | 提取实际 x 坐标确认位置 |
| 忽略坐标提取 | 只记录文本内容，不核对位置 | 必须获取并核对 x 坐标 |

## 2.2 文字、图片、布局、层级（必记）

### 文字

区域内每一处可见文案都要记录。

**记录项**：
- **内容**（完整或典型示例）
- **字体**（fontFamily）
- **字号**（fontSize）
- **字重**（fontWeight）
- **颜色**（color，含透明度）
- **行高**（lineHeight）
- **字间距**（letterSpacing，若有）

**易漏点**：
- 页面最上方的欢迎语/大标题
- 副标题
- 按钮/链接文案
- 占位提示文字
- Tab 或标签文字

**检查方法**：按「从上到下」扫到时立即记下。

### 图片

区域内每一处图片、占位图、图标区都要记录。

**记录项**：
- **尺寸**（宽×高）
- **位置**（相对父级或相邻元素）
- **比例/圆角**
- **与相邻元素的间距**
- **多图拼接时的布局**（左右/上下/网格）及每张的占比

**易漏点**：
- 小图标
- 占位图
- 多图拼接时的布局关系

**检查方法**：逐区域扫描，确保每张图片都有记录。

### 布局

位置、尺寸、间距（padding/gap/margin）、对齐方式、排列方向（横/纵/网格）、多图拼接方式等必须与设计稿一致，不可错位或比例失调。

**记录项**：
- 位置（x, y）
- 尺寸（width, height）
- 间距（padding, gap, margin，含方向）
- 对齐方式（align-items, justifyContent）
- 排列方向（横向/纵向/网格）
- 多图拼接方式

**检查方法**：对照设计稿，确保布局参数与设计稿一致。

### 层级

从外到里的嵌套（谁包谁）、父子/兄弟关系、区块先后与叠放顺序必须写清，实现时不可丢层或顺序颠倒。

**⚠️ 强制要求：必须使用 MCP 工具读取实际节点树**

**禁止凭视觉推断层级结构！** 层级必须从设计稿工具返回的实际节点数据中精确提取。

**使用 MCP 工具提取层级**：

**MasterGo**：
```python
# 递归提取节点树
def extract_hierarchy(node, parent_x=0, parent_y=0, depth=0):
    layout = node.get('layoutStyle', {})
    rel_x = layout.get('relativeX') or 0
    rel_y = layout.get('relativeY') or 0
    abs_x = parent_x + rel_x
    abs_y = parent_y + rel_y

    result = {
        'type': node.get('type', ''),
        'name': node.get('name', ''),
        'x': abs_x,
        'y': abs_y,
        'width': layout.get('width') or 0,
        'height': layout.get('height') or 0,
        'children': []
    }

    # 递归提取子节点
    for child in node.get('children', []):
        result['children'].append(extract_hierarchy(child, abs_x, abs_y, depth + 1))

    return result
```

**Pencil**：
```javascript
// 读取区域及其子节点
mcp__pencil__batch_get({
  filePath: "<.pen 文件路径>",
  nodeIds: ["<区域节点ID>"],
  readDepth: 5  // 读取足够深的层级
})
```

**记录项**：
- 嵌套关系（谁包谁、父子/兄弟）
- 先后顺序
- 叠放关系（z-index）
- 每个子节点的坐标和尺寸

**检查方法**：
1. **必须使用 MCP 工具读取区域的实际 children 数据**
2. **按实际节点树记录层级**，而非凭视觉推断
3. **核对每个子节点的坐标是否在父区域内**

**常见层级推断错误**：
| 错误类型 | 错误示例 | 正确做法 |
|----------|----------|----------|
| 图例位置推断 | "图例应该在标题下面" | 读取实际节点，图例可能在图表区域内 |
| 统计卡片区推断 | "营收平衡是底部统计区" | 读取实际节点，可能是左侧独立卡片组 |
| 图表类型推断 | "这是双轴柱状图" | 读取节点名称，实际可能是堆叠面积图 |

## 2.4 元素层级与嵌套（层级必准确）

### 从外到里

按「**从外到里**」：外层容器 → 内层容器 → 叶子元素；**层级是四类重中之重之一，嵌套与先后顺序不可错。**

### 卡片类

区分「整卡」与「卡内区块」（如标题区、图片区、操作区），每块内的文字、图片、布局、层级单独列清。

**示例**：
```
卡片容器
├── 标题区
│   ├── 标题文字
│   └── 副标题文字
├── 图片区
│   └── 图片
└── 操作区
    ├── 按钮1
    └── 按钮2
```

### 列表/网格

明确单项尺寸、间距、每行/列数量，以及单项内的文字、图片与层级关系。

**示例**：
```
列表容器
├── 列表项1
│   ├── 图标
│   ├── 标题文字
│   └── 描述文字
├── 列表项2
│   └── ...
└── 列表项3
    └── ...
```

### 表格区域（特殊要求）

表格是高频错误区，必须逐列提取，**禁止推断**。

**提取要求**：
1. **逐列读取**：使用设计稿工具获取每一列的节点，读取实际的 `content` 文字
2. **禁止经验推断**：不得根据页面名称（如"项目情况"）假设表格列名
3. **列数确认**：确认总列数，包括无标题的复选框列
4. **列宽记录**：记录每列的精确宽度（带单位）
5. **特殊列标注**：含头像、进度条、标签、状态指示器的列需特别标注
6. **表头层级**：确认是否有分组表头（多层级表头）

**提取示例**：
```
表格区域
├── 表格列头（共8列）
│   ├── 第1列：复选框（32px，无标题文字）
│   ├── 第2列：重点项目（160px）
│   ├── 第3列：项目名称（306px）
│   ├── 第4列：状态（152px）
│   ├── 第5列：当月评级（125px，含头像+评级）
│   ├── 第6列：交付截止日（222px）
│   ├── 第7列：当前里程碑（292px）
│   └── 第8列：预算执行比例（292px，含进度条+百分比）
└── 表格行数据
    └── ...
```

**警示**：
- ❌ 禁止：「这是一个项目表格，应该有项目编号、项目名称、交付部门等列」
- ✅ 正确：「读取到第2列的实际文字是'重点项目'，第5列是'当月评级'...」

## 2.5 常见遗漏检查点（分析时必过一遍）

详见 `checklist-common-misses.md` 获取完整检查清单。

快速检查：

- [ ] **所有可见文字**：欢迎语、大标题、副标题、说明、按钮/链接/占位/Tab 文案等，按从上到下逐条核对
- [ ] **所有图片/占位图**：尺寸、位置、比例、多图拼接布局，无遗漏
- [ ] **布局**：位置、尺寸、间距、对齐、排列方向、多图拼接方式与设计稿一致，无错位或比例失调
- [ ] **层级**：从外到里的嵌套与先后顺序正确，无丢层或顺序颠倒
- [ ] **⚠️ 子元素顺序**：横向排列的子元素必须按 x 坐标排序，不得按树结构顺序记录
- [ ] 背景色（页面、卡片、区块、按钮）
- [ ] 边框（颜色、宽度、圆角）
- [ ] 阴影（颜色、模糊、偏移）
- [ ] 内边距与外边距（padding / gap / margin）
- [ ] 对齐方式（左对齐 / 居中 / 右对齐）
- [ ] **右侧或对称区域**（易只写左侧漏右侧）
- [ ] **底部或折叠区域**
- [ ] 小图标、小按钮、Tab 文字
- [ ] 占位图/占位图标尺寸与比例
