# 校验清单

## 执行时机

**文档生成完成后，必须执行以下校验步骤。**

> **⚠️ 校验前必须重新获取设计稿数据**
>
> 所有校验必须基于最新设计稿数据进行，禁止仅凭记忆或缓存数据校验。

---

## 校验零：区域数量校验（必须首先执行）

### 目的

确保文档记录的区域数量与设计稿实际区域数量完全一致。

### 步骤

1. **统计设计稿顶层节点数量**
   - 使用工具获取根节点的所有直接子节点
   - 过滤出 GROUP/FRAME/INSTANCE 类型且有实际尺寸（w > 100, h > 50）的节点

2. **统计文档区域列表数量**
   - 计算「一、设计稿整体分析」中「区域列表」表格的行数

3. **数量对比**

### 校验表格

```markdown
### 区域数量校验

| 项目 | 数量 | 是否一致 |
|------|------|----------|
| 设计稿顶层区域数 | ___ 个 | - |
| 文档区域列表行数 | ___ 行 | ✅ / ❌ |

**若不一致**：
- [ ] 已逐个对比，找出遗漏区域
- [ ] 已补充遗漏区域到文档
```

## 校验一：区域顺序校验

### 目的

确保区域列表按「从上到下、从左到右」排序。

### 步骤

1. 获取设计稿所有区域的 x、y 坐标
2. 首先按 y 排序，y 相同时按 x 排序
3. 对比文档顺序

### 校验表格

```markdown
### 区域顺序校验

| 序号 | 区域名称 | 设计稿 y | 设计稿 x | 文档顺序 | 是否正确 |
|------|----------|----------|----------|----------|----------|
| 1    | 导航栏   | 0        | 0        | 第1位    | ✅       |
```

## 校验六：表格列标题专项校验（必须严格执行）

### 目的

确保表格列标题与设计稿实际内容完全一致，禁止根据经验推断。

### ⚠️⚠️⚠️ 真实遗漏案例（2026-04-08）

**案例：遗漏「项目名称」列**

| 问题 | 原因 | 后果 |
|------|------|------|
| 「项目名称」列未记录 | 只凭部分数据推断，未系统遍历所有列 | 文档列数错误，前端开发遗漏必要功能 |

**错误过程**：
1. 读取表格列节点 → 看到 10 个列节点
2. 部分表头内容是占位符「Title」→ 凭经验跳过
3. 未逐列读取每个表头的实际内容 → 遗漏「项目名称」列

**教训**：必须系统性遍历所有列节点，不能凭经验跳过！

### 🔴🔴🔴 强制步骤：表格列系统性遍历

**必须按以下步骤逐一分析，禁止跳过任何列！**

```markdown
### 表格列遍历清单（校验阶段必填）

步骤1：获取表格 columns 节点的所有子节点
- 列节点总数：___ 个

步骤2：逐列遍历（每个列节点必须执行）
| 列序号 | 列ID | enabled | 表头内容 | 列宽 | 是否记录 |
|--------|------|---------|----------|------|----------|
| 1 | ___ | true/false | ___ | ___ px | ✅/❌禁用 |
| 2 | ___ | true/false | ___ | ___ px | ✅/❌禁用 |
| ... | ... | ... | ... | ... | ... |

步骤3：验证列宽总和
- 列宽总和：___ px
- 表格宽度：___ px
- 差值：___ px（应接近0）
```

### 步骤

1. **获取列节点列表**：读取表格 columns 节点的所有 children
2. **统计列总数**：记录列节点总数
3. **逐列遍历**：对每个列节点执行：
   - 检查列节点的 `enabled` 字段
   - 读取列节点的 width 属性
   - 读取第一个子节点（表头单元格）
   - 读取表头单元格内的文字节点内容
4. **验证数量**：列宽总和应接近表格宽度
5. **筛选可见列**：最终文档只记录 `enabled: true` 或无 enabled 字段的列

### 关键字段

| 字段 | 值 | 状态说明 | 对设计稿的影响 |
|------|-----|----------|----------------|
| `enabled` | `false` | 禁用状态 | **列不显示或不可交互** |
| `enabled` | `true` | 启用状态 | 正常显示 |

### 列类型识别

| 列类型 | 识别特征 | 说明 |
|--------|----------|------|
| checkbox | 无标题文字，宽度约32px | 复选框列 |
| text | 有标题文字 | 文字列 |
| avatar | 列标题可能为「头像」或无标题 | 头像列 |
| actions | 列标题可能为「操作」或无标题 | 操作列 |

### ⚠️ 真实遗漏案例（2026-04-07）

**案例：表格禁用列被误认为可见**

| 问题 | 文档记录 | 设计稿实际 |
|------|----------|------------|
| Checkbox列 | 认为可见 | `enabled: false`，不可见 |
| 操作列 | 认为可见 | `enabled: false`，不可见 |
| 「类型」列 | 文档记录存在 | 设计稿没有此列 |
| 「进度」列 | 文档记录存在 | 设计稿没有此列 |

**后果**：前端开发可能实现不需要的功能，遗漏实际存在的列

### 校验表格（只验证可见列）

```markdown
### 表格列标题专项校验

| 列序号 | 设计稿列标题 | 文档记录 | 是否一致 |
|--------|--------------|----------|----------|
| 1 | 重点项目 | 重点项目 | ✅ |
| 2 | 项目名称 | 项目名称 | ✅ |
| 3 | 状态 | 状态 | ✅ |

> 禁用列已排除，只验证可见列。
```

### 强制规则

- [ ] **必须获取表格 columns 节点的完整 children 列表**
- [ ] **必须对每个列节点逐一读取，禁止凭经验跳过**
- [ ] **必须读取每个列节点的表头单元格内文字，不能凭列名推断**
- [ ] **表头内容是占位符（如「Title」）时，仍需记录该列存在**
- [ ] **必须检查每个列节点的 `enabled` 字段，不是表头单元格的字段**
- [ ] **必须验证列宽总和 ≈ 表格宽度**
- [ ] **每列必须检查 enabled 字段**
- [ ] **enabled: false 的列不在文档中记录**
- [ ] **文档记录的列数必须与 enabled: true 的列数一致**

## 校验九：子元素归属校验（必须严格执行）

### 目的

确保文档中记录的子元素确实属于该区域。

### 步骤

1. 使用工具读取区域的实际 children 数据
2. 计算区域边界
3. 验证每个子元素坐标是否在范围内

### 验证公式

```
子元素.x >= 父区域.x
子元素.x + 子元素.width <= 父区域.x + 父区域.width
子元素.y >= 父区域.y
子元素.y + 子元素.height <= 父区域.y + 父区域.height
```

### 校验表格

```markdown
### 子元素归属校验

| 区域名称 | 区域边界 | 文档记录子元素 | 子元素坐标 | 是否在范围内 | 验证结果 |
|----------|----------|----------------|------------|--------------|----------|
| 卡片A | 160, 112, 522, 424 | Tab切换 | 1486, 128 | ❌ 超出边界 | 归属错误 |
```

## 校验十四：嵌套容器同名校验（必须严格执行）

### 目的

检测父子节点或兄弟节点同名的情况，避免遗漏嵌套内容。

### ⚠️ 真实遗漏案例（2026-04-02）

**案例：项目详情弹窗遗漏「第二行信息区」**

```
Group 220（基础信息区）
├── Group 219（第一行，y=20）← 读到了
└── Group 220（第二行，y=94）← 名字与父节点相同，被忽略了！
```

**遗漏内容**：
- 约定交付截止日
- 确收日期
- 业务经理

**教训**：同名容器非常隐蔽，必须检查所有子节点的 y 坐标跳跃。

### 检查方法

1. **读取完整 children 列表**：不只看名称，要看坐标
2. **检测 y 坐标跳跃**：如果同一容器内有两个子节点 y 差值 > 20px，可能存在多行
3. **统计子节点数量**：设计稿实际子节点数 vs 文档记录数

### 校验表格

```markdown
### 嵌套容器同名校验

| 容器名称 | 子节点数量 | 子节点 y 坐标列表 | 是否有多行 | 文档记录行数 |
|----------|------------|-------------------|------------|--------------|
| 基础信息区 | 2 | 20, 94 | ✅ 有2行 | 2行 |
```

### 强制规则

- [ ] **每个容器必须检查所有直接子节点**
- [ ] **发现同名节点时，用坐标区分**
- [ ] **y 坐标跳跃 > 20px 时，必须检查是否有多行内容**

## 校验十五：信息行/列数量校验（必须严格执行）

### 目的

确保多行/多列信息区域的所有行、列都已记录，没有遗漏。

### 常见遗漏场景

| 场景 | 遗漏原因 | 示例 |
|------|----------|------|
| 多行信息区 | 只记录了第一行 | 第一行4列 + 第二行3列，只记录了第一行 |
| 嵌套标签 | 标签在数值组内部被忽略 | 「进度概况」标签嵌套在日期值组内 |
| 隐藏列 | 列宽较窄或无标题 | 复选框列、操作列 |

### 检查方法

1. **统计设计稿信息行数**：
   ```
   找出所有 y 坐标不同的子容器
   按 y 排序后统计行数
   ```

2. **统计每行列数**：
   ```
   找出每行内的子节点数量
   对比文档记录
   ```

3. **检查嵌套标签**：
   ```
   不只看主要数值，还要看数值组内部的标签
   ```

### 校验表格

```markdown
### 信息行/列数量校验

| 区域名称 | 设计稿行数 | 文档记录行数 | 每行列数 | 是否一致 |
|----------|------------|--------------|----------|----------|
| 基础信息区 | 2行 | 2行 | 4列+3列 | ✅ |
| 项目进度 | 1行 | 1行 | 3列 | ✅ |
```

### 强制规则

- [ ] **统计信息区域总行数**

## 校验十六：深度遍历完整性校验（必须严格执行）

### 目的

确保读取深度足够，没有因为深度限制而遗漏内容。

### 问题原因

| 读取深度 | 可能遗漏的内容 |
|----------|----------------|
| depth=1 | 所有嵌套结构 |
| depth=2 | 第二层容器内的内容 |
| depth=3 | 第三层详情（如数值组内的标签） |

### ⚠️ 真实遗漏案例

**案例：读取深度不足导致遗漏**

```javascript
// ❌ 错误：只读到 depth 2
batch_get({ nodeIds: ["CHjBi"], readDepth: 2 })

// ✅ 正确：读到 depth 4，并展开实例
batch_get({
  nodeIds: ["CHjBi"],
  readDepth: 4,
  resolveInstances: true
})
```

### 推荐读取深度

| 设计复杂度 | 推荐 readDepth | 说明 |
|------------|----------------|------|
| 简单页面 | 2 | 单层结构 |
| 中等复杂 | 3 | 有嵌套容器 |
| 复杂页面 | 4+ | 多层嵌套、实例组件 |

### 强制规则

- [ ] **初次读取使用 readDepth: 4**
- [ ] **发现嵌套结构时，对嵌套容器单独读取**
- [ ] **使用 resolveInstances: true 展开组件实例**

## 校验二十：子元素坐标归属校验（必须严格执行）

### ⚠️ 重要提示：此校验必须在分析阶段执行！

**不是校验阶段才发现问题，而是分析阶段就必须验证！**

**🔴🔴🔴 前置条件：必须先完成「区域边界验证表」！**

**错误做法**：分析时凭直觉判断 → 写入文档 → 校验时才发现错误
**正确做法**：第一步记录边界 → 分析时验证坐标 → 确认归属 → 写入文档

### 目的

确保文档中记录的子元素归属与设计稿实际坐标一致，防止元素归属错误。

**⚠️ 强制要求：禁止凭视觉直觉判断元素归属！必须验证坐标边界！**

### ⚠️ 真实遗漏案例（2026-04-08）

**案例：年份选择器归属错误**

| 问题 | 设计稿坐标 | 错误归属 | 正确归属 |
|------|------------|----------|----------|
| 年份选择器 | x=546 | 右上区域(x≥698) | 左侧卡片(160≤x≤682) |

**验证过程**：
- 左侧卡片区：x=160, width=522, 右边界=160+522=682
- 年份选择器 x=546
- 验证：160 ≤ 546 < 682 ✅ → 属于左侧卡片区
- 右侧图表区：x=698
- 验证：546 < 698 ❌ → 不属于右侧图表区

**原因分析**：
1. **凭视觉直觉判断**：看到年份选择器在右侧附近，直觉认为控制右侧图表
2. **未验证坐标边界**：没有计算区域边界值并验证元素坐标
3. **左右布局认知偏差**：认为左右布局的元素一定归属各自的区域

**后果**：文档将年份选择器错误归属到右侧图表区，导致开发实现时组件位置错误

### 检查方法

#### 强制检查步骤

1. **记录区域边界**：分析每个区域时，必须记录 x、y、width、height
2. **计算边界值**：
   - 左边界 = x
   - 右边界 = x + width
   - 上边界 = y
   - 下边界 = y + height
3. **验证每个元素归属**：
   - 元素 x 坐标必须在区域 x 范围内
   - 元素 y 坐标必须在区域 y 范围内

#### 验证公式

```
元素归属正确的条件：
  区域.x ≤ 元素.x < 区域.x + 区域.width

或简化判断：
  元素.x >= 区域左边界 AND 元素.x < 区域右边界
```

### 校验表格

```markdown
### 子元素坐标归属校验

| 元素名称 | 元素 x 坐标 | 归属区域 | 区域左边界 | 区域右边界 | 是否在范围内 | 验证结果 |
|----------|-------------|----------|------------|------------|--------------|----------|
| 年份选择器 | 546 | 左侧卡片 | 160 | 682 | ✅ 160≤546<682 | 正确 |
| Tab切换 | 1486 | 右侧图表 | 698 | 1760 | ✅ 698≤1486<1760 | 正确 |
```

### 左右布局边界验证（重点）

**左右布局中，边界附近的元素必须重点验证！**

| 元素 x 坐标位置 | 验证重点 | 常见错误 |
|-----------------|----------|----------|
| x 在左区域右边界±50px | 必须验证是否在左区域内 | 容易误判到右区域 |
| x 在右区域左边界±50px | 必须验证是否在右区域内 | 容易误判到左区域 |
| x 跨越两个区域 | 必须明确说明归属 | 未验证直接判断 |

### 强制规则

- [ ] **分析阶段必须记录每个区域的边界值（左边界、右边界）**
- [ ] **每个子元素归属必须验证 x 坐标是否在区域边界内**
- [ ] **左右布局中，边界附近（x 在边界±50px 范围）的元素必须重点验证**
- [ ] **层级结构中，必须标注独立节点的坐标位置**
- [ ] **禁止凭视觉直觉判断元素归属**
- [ ] **发现坐标归属问题后，必须立即修正文档中的层级结构（文字、图片、层级三个部分都要修正）**

## 校验二十二：INSTANCE 组件标题提取校验（MasterGo 必须严格执行）

### 目的

确保从 MasterGo INSTANCE 组件内部正确提取标题文字，避免标题不一致或遗漏。

### ⚠️ 真实遗漏案例（2026-04-08）

**案例：MasterGo INSTANCE 组件标题未正确提取**

| 问题 | 文档记录标题 | MasterGo 实际标题 | 原因 |
|------|--------------|-------------------|------|
| 区域3 | 营收平衡趋势 | **项目营收平衡情况** | 未读取 INSTANCE 内标题 |
| 区域4 | 人员评级分布 | **部门成员** | 未读取 INSTANCE 内标题 |
| 区域5 | 项目情况 | **部门项目情况** | 未读取 INSTANCE 内标题 |

**根本原因**：
1. MasterGo 标题文字存储在 INSTANCE 组件内部，不在顶层 GROUP 节点
2. INSTANCE 节点的 children 包含实际的标题 TEXT 节点
3. 常规深度遍历可能跳过 INSTANCE 内部结构

### MasterGo 节点结构示例

```
GROUP (组14785, y=552) - 区域容器
├── INSTANCE (id: 16:1519) - 标题组件 ⬅️ 关键！
│   └── GROUP (组14765)
│       ├── TEXT「部门成员」⬅️ 实际标题在这里！
│       └── SVG_ELLIPSE (圆形图标)
├── GROUP (图表/环形图)
└── FRAME (图例区)
```

### 强制检查步骤

1. **识别 INSTANCE 节点**
   ```
   遍历区域容器的直接子节点
   找出 type === 'INSTANCE' 的节点
   ```

2. **递归读取 INSTANCE 内部**
   ```
   对 INSTANCE 节点执行深度遍历
   查找 name === '标题' 或 y < 30 的 TEXT 节点
   提取实际文字内容
   ```

3. **验证标题一致性**
   ```
   对比文档记录的标题与实际提取的标题
   不一致时必须修正
   ```

### Python 代码示例

```python
def extract_instance_title(instance_node):
    """从 INSTANCE 组件内部提取标题文字"""
    children = instance_node.get('children', [])

    def find_title_text(nodes, depth=0, max_depth=3):
        for node in nodes:
            if node.get('type') == 'TEXT':
                node_name = node.get('name', '')
                text_raw = node.get('text', '')
                y = node.get('layoutStyle', {}).get('relativeY', 0)

                # 提取实际文字
                text = extract_text_content(text_raw)

                # 标题特征：名称为「标题」或 y 坐标较小
                if node_name == '标题' or (y < 30 and text and len(text) > 2):
                    return text

            # 递归查找
            if node.get('children') and depth < max_depth:
                result = find_title_text(node['children'], depth+1, max_depth)
                if result:
                    return result
        return None

    return find_title_text(children)
```
